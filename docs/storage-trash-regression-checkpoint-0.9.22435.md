# 0.9.22435 저장소/휴지통 회귀 테스트 1차

## 목적

작업지시서 lazy load 1차 안정화 이후 `/admin/files` 저장소 관리와 휴지통 흐름이 기존 정책대로 유지되는지 확인한다. 이번 버전은 저장소/휴지통 회귀 테스트 기준을 고정하는 체크포인트이며, DB schema와 R2 Worker 동작은 변경하지 않는다.

## 현재 정책 기준

1. 고객관리자 `/admin/files` 휴지통은 복원 가능한 항목 중심으로 표시한다.
2. 고객관리자가 선택 삭제한 항목은 실제 R2 삭제가 아니라 삭제 요청 상태로 전환한다.
3. `purge_requested` 항목은 고객관리자 휴지통 기본 목록에서 숨기고 `/system/storage-usage` 실제 삭제 후보에서 처리한다.
4. 작업지시서 삭제 요청은 작업지시서 대표 row와 문서/디자인/메모 묶음을 기준으로 처리한다.
5. 작업지시서 복원은 작업지시서와 함께 삭제된 문서/디자인/메모를 함께 복원한다.
6. 개별 문서/디자인/메모는 부모 작업지시서가 삭제되지 않은 경우에만 파일 단위 복원/삭제 요청이 가능하다.
7. 고객관리자 화면에서는 `연결 첨부`, `영구삭제`, `작지`, `복구` 표현을 사용하지 않는다.
8. 고객관리자 메시지는 `작업지시서 n건과 문서 n개, 디자인 n개, 메모 n개`처럼 역할 기반 항목명으로 표시한다.
9. R2 실제 삭제는 시스템관리자 Worker 기반 purge 흐름에서만 실행한다.

## 로컬 사전 준비

개발 DB를 초기화하고 통계/첨부 테스트 데이터를 다시 넣을 때는 아래 순서를 사용한다.

```powershell
cd C:\CWJ_Project\peacebypiece-2.0

psql $env:DATABASE_URL -f db/schema/full_reset.sql
psql $env:DATABASE_URL -f db/schema/full_reset_smoke_test.sql
psql $env:DATABASE_URL -f db/schema/seed_realistic_workorders_0_9_2227.sql
```

R2 더미 파일이 필요한 경우에는 통계 fixture 기준으로 먼저 계획을 확인한 뒤 업로드한다.

```powershell
node scripts/seed-r2-demo-files.mjs --preset=small --mode=plan --only-stats-fixtures
node scripts/seed-r2-demo-files.mjs --preset=small --mode=all --confirm-upload --only-stats-fixtures
node scripts/seed-r2-demo-files.mjs --preset=small --mode=verify --only-stats-fixtures
```

## 테스트 1 — 문서 파일 단독 삭제

1. 작업지시서 상세에서 문서 첨부가 있는 작업지시서를 연다.
2. 문서 첨부를 삭제한다.
3. `/admin/files` 휴지통에서 해당 문서가 표시되는지 확인한다.
4. 작업지시서 대표 row에 묶여 있는 항목이 아니라 개별 파일 row인지 확인한다.
5. 문구가 `문서` 기준으로 표시되는지 확인한다.

성공 기준:

- 문서 row가 휴지통에 표시된다.
- 부모 작업지시서가 활성 상태라면 파일 단위 복원/선택 삭제가 가능하다.
- 고객관리자 화면에 `영구삭제`, `연결 첨부` 표현이 나오지 않는다.

## 테스트 2 — 디자인 파일 단독 삭제

1. 작업지시서 상세에서 디자인 첨부가 있는 작업지시서를 연다.
2. 디자인 첨부를 삭제한다.
3. `/admin/files` 휴지통에서 해당 디자인이 표시되는지 확인한다.
4. 미리보기 썸네일이 실패해도 휴지통 row와 액션이 깨지지 않는지 확인한다.

성공 기준:

- 디자인 row가 `디자인` 기준으로 표시된다.
- 썸네일/R2 preview 실패가 복원/삭제 요청 액션을 막지 않는다.

## 테스트 3 — 메모 단독 삭제

1. 메모가 있는 작업지시서를 연다.
2. 메모를 삭제한다.
3. `/admin/files` 또는 작업지시서 휴지통 정책상 메모가 표시되는 위치를 확인한다.
4. 메모 수가 작업지시서 묶음 요약에 포함되는지 확인한다.

성공 기준:

- 메모는 문서/디자인과 분리되어 `메모 n개`로 집계된다.
- 메모를 문서나 디자인으로 잘못 계산하지 않는다.

## 테스트 4 — 파일 단독 복원

1. `/admin/files` 휴지통에서 부모 작업지시서가 활성 상태인 문서 또는 디자인을 선택한다.
2. 복원 버튼을 실행한다.
3. 성공 메시지와 목록 갱신을 확인한다.
4. 작업지시서 상세를 다시 열어 해당 첨부가 보이는지 확인한다.

성공 기준:

- 복원 메시지는 `문서 n개`, `디자인 n개` 기준으로 표시된다.
- 복원된 파일은 휴지통에서 사라진다.
- 작업지시서 상세에서 다시 확인된다.

## 테스트 5 — 파일 단독 삭제 요청

1. `/admin/files` 휴지통에서 부모 작업지시서가 활성 상태인 문서 또는 디자인을 선택한다.
2. 선택 삭제를 실행한다.
3. 고객관리자 휴지통 기본 목록에서 해당 항목이 사라지는지 확인한다.
4. `/system/storage-usage` 실제 삭제 후보에 해당 첨부가 표시되는지 확인한다.

성공 기준:

- 고객관리자 화면에서는 실제 R2 삭제가 일어나지 않는다.
- 삭제 요청 결과 메시지는 `문서 n개`, `디자인 n개` 기준으로 표시된다.
- 시스템관리자 실제 삭제 후보로 넘어간다.

## 테스트 6 — 작업지시서 삭제

1. 작업지시서 상세 또는 목록에서 작업지시서를 삭제한다.
2. `/admin/files` 휴지통에서 작업지시서 대표 row가 표시되는지 확인한다.
3. 해당 작업지시서에 연결된 문서/디자인/메모가 대표 row 아래 묶음 row로 표시되는지 확인한다.
4. 묶음 row는 개별 선택 가능한 파일처럼 보이지 않는지 확인한다.

성공 기준:

- 작업지시서 row는 작업지시서 단위로 구분된다.
- 함께 삭제된 문서/디자인/메모는 묶음 항목으로 표시된다.
- 묶음 항목은 작업지시서 단위 복원/삭제 요청 정책을 따른다.

## 테스트 7 — 작업지시서 복원

1. `/admin/files` 휴지통에서 작업지시서 대표 row를 선택한다.
2. 복원 범위 확인 모달을 연다.
3. 문서/디자인/메모 개수와 용량 요약을 확인한다.
4. 복원을 실행한다.
5. 작업지시서와 함께 삭제된 문서/디자인/메모가 함께 복원되는지 확인한다.

성공 기준:

- 결과 메시지는 `작업지시서 n건과 문서 n개, 디자인 n개, 메모 n개를 복원하였습니다.` 계열로 표시된다.
- 복원 후 작업지시서가 목록/상세에서 다시 접근 가능하다.
- 연결 항목이 휴지통에 남지 않는다.

## 테스트 8 — 작업지시서 삭제 요청

1. `/admin/files` 휴지통에서 작업지시서 대표 row를 선택한다.
2. 선택 삭제 범위 확인 모달을 연다.
3. 삭제 요청을 실행한다.
4. 고객관리자 휴지통 기본 목록에서 작업지시서 대표 row와 묶음 항목이 사라지는지 확인한다.
5. `/system/storage-usage`에서 실제 삭제 후보로 집계되는지 확인한다.

성공 기준:

- 작업지시서는 `purge_requested` 상태로 전환된다.
- 문서/디자인/메모는 시스템관리자 실제 삭제 후보로 넘어간다.
- R2 실제 삭제는 고객관리자 화면에서 실행되지 않는다.

## 테스트 9 — 휴지통 비우기

1. 복원 가능한 휴지통 항목이 있는 상태에서 비우기 버튼을 누른다.
2. 확인 모달에서 실행한다.
3. 파일 단독 항목과 작업지시서 대표 row가 함께 선택 삭제 요청 처리되는지 확인한다.
4. 고객관리자 휴지통 기본 목록이 비워지는지 확인한다.
5. `/system/storage-usage` 후보 수와 용량이 증가하는지 확인한다.

성공 기준:

- 휴지통 비우기는 실제 R2 삭제가 아니라 삭제 요청 처리다.
- 작업지시서 자체 개수는 고객관리자 파일 운영 요약의 삭제 요청 개수에 합산하지 않는다.
- 첨부파일 개수와 용량만 삭제 요청 요약에 합산한다.

## 테스트 10 — 금지 문구 확인

아래 문자열이 고객관리자 화면에 노출되지 않아야 한다.

```text
연결 첨부
영구삭제
작지
복구
```

권장 표현:

```text
문서
디자인
메모
복원
삭제 요청
선택 삭제
작업지시서
```

## 테스트 11 — API 확인 명령

파일 단독 휴지통 snapshot 확인:

```powershell
Invoke-RestMethod -Uri "http://localhost:3000/api/admin/files/snapshot" -Method GET |
ConvertTo-Json -Depth 10 |
Out-File -Encoding utf8 .tmp\admin-files-snapshot-0.9.22435.json
```

시스템관리자 실제 삭제 후보 확인:

```powershell
Invoke-RestMethod -Uri "http://localhost:3000/api/admin/files/trash/purge-candidates?limit=50" -Method GET |
ConvertTo-Json -Depth 10 |
Out-File -Encoding utf8 .tmp\admin-files-purge-candidates-0.9.22435.json
```

Worker dry-run 확인:

```powershell
Invoke-RestMethod -Uri "http://localhost:3000/api/admin/files/trash/purge-worker?dryRun=true&limit=50" -Method POST |
ConvertTo-Json -Depth 10 |
Out-File -Encoding utf8 .tmp\admin-files-purge-worker-dry-run-0.9.22435.json
```
