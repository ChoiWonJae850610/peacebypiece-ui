# 0.10.45 — 기준정보 fallback 혼입 제거와 생산품 유형 저장 보정

## 목적

고객관리자 환경설정의 기준정보 수량이 fallback 데이터와 DB 데이터 사이에서 흔들리는 문제를 줄이고, 생산품 유형 기본값 복원 후 저장이 실제 고객사 `item_categories` 데이터로 유지되도록 보정한다.

## 변경 기준

- DB가 설정된 상태에서는 기준정보 카드 숫자를 DB 조회 결과 기준으로 표시한다.
- fallback 데이터는 DB 미설정 또는 API 오류 상황의 임시 표시 용도로만 사용한다.
- 생산품 유형 기본값 복원은 고객사 `item_categories`를 실제로 교체 저장한다.
- 단위 표준과 외주공정 유형은 시스템 표준 원장 + 고객사별 사용 여부 구조를 유지한다.

## 생산품 유형 저장 보정

기존 방식은 기본값 복원으로 들어온 항목을 upsert한 뒤 남은 항목을 삭제하는 흐름이었다. 이 경우 기존 고객사 데이터와 기본값 데이터가 같은 이름을 다른 id로 갖고 있으면 `company_id, parent_id, name` 고유 제약과 충돌할 수 있다.

0.10.45에서는 생산품 유형 저장 시 고객사 기준 `item_categories`를 먼저 삭제한 뒤 1차, 2차, 3차 순서로 다시 삽입한다. 개발 중인 현재 정책에서는 이 방식이 기본값 복원과 전체 교체 저장에 더 명확하다.

## fallback 표시 원칙

- `/api/admin/standards`가 DB 모드에서 빈 배열을 반환하면 빈 상태로 표시한다.
- 화면에서 임의로 기본값 개수를 섞어 표시하지 않는다.
- 작업지시서 단위/단가 기준 선택지는 DB 모드에서 사용 단위가 없으면 빈 선택지로 둔다.
- DB 미설정 또는 API 오류 상황에서는 기존 fallback 선택지를 사용할 수 있다.

## 후속 작업

- 작업지시서 기준정보 선택지 저장값 호환성 점검
- 시스템 표준 원장 seed/초기화 흐름 점검
- 신규 고객사 생성 시 생산품 유형 기본 템플릿 복사 로직 설계
