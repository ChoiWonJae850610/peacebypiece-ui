# 0.24.34.7 Refactor Candidates Audit

Status: AUDIT ONLY  
Baseline: 0.24.34.6  
Result version: 0.24.34.7  
Scope: Identify refactor candidates without modifying business behavior

## 1. Refactor rule

Do not combine broad refactoring with product-critical work such as PDF generation, size standards, signup approval, billing, storage deletion, or export.

Refactors should be small, route-bounded, and evidence-backed.

## 2. Large file candidates

The following areas are candidates for future decomposition:

- `tools/pipeline/peacebypiece-auto-pipeline.ps1`
- `app/ui/WaflUiCatalogPage.tsx`
- `lib/i18n/ko/admin.ts`
- `lib/i18n/en/admin.ts`
- `lib/invitations/joinRequestRepository.ts`
- `components/admin/settings/AdminSettingsHub.tsx`
- `components/admin/members/AdminMemberManagementDashboard.tsx`
- `lib/workorder/api/workOrderRouteHandlers.ts`

Recommended split direction:

- PowerShell pipeline: menu handlers / ZIP export / repo-state / verification profiles / helper functions
- WAFL catalog: section registry and per-component example modules
- i18n: domain namespaces by route or feature
- workorder route handlers: action-specific modules
- admin dashboards: screen sections and table/action components

## 3. WAFL component cleanup candidates

Route/screen files should avoid direct local implementations of:

- modal overlay
- drawer overlay
- raw buttons
- raw inputs
- raw selects
- ad-hoc tables
- local status badges
- repeated empty/loading/error states

Known areas to review:

- signup application dashboard
- system signup review screens
- system company console
- category rule mobile drawer
- workorder drawing/canvas editor
- dev/test modals
- admin onboarding gates

Policy:

- Common WAFL components may use low-level HTML internally.
- Screen components should prefer WAFL components.
- Do not replace all raw elements globally; audit by route and test after each route.

## 4. Internal term cleanup candidates

Rendered UI should avoid raw/internal terms such as:

- `provisioning_failed`
- `payment_readiness`
- `consent evidence`
- raw enum names
- untranslated English system labels
- mock/fake/sample wording outside dev-only areas

Policy:

- Dev/test routes may expose technical language if clearly gated.
- Customer and system-admin production surfaces should use Korean product copy.
- Do not blindly replace terms in code identifiers or docs.

## 5. PDF builder consolidation candidates

Workorder PDF and supplier/order-request PDF must remain distinct products.

Allowed:

- shared layout primitives
- shared representative image handling
- shared table styling utilities
- shared PDF download/view plumbing

Not allowed:

- merging factory workorder PDF with supplier/order-request PDF semantics
- leaking unit price/amount/cost/margin/loss into factory-delivery workorder PDF

## 6. Refactor timing

Before 0.24.35 Export:

- only low-risk pipeline/documentation cleanup
- only route-bounded UI cleanup necessary for PDF/size/signup/dashboard continuation

After 0.24.34.x productization checkpoints:

- split large files
- normalize route-level WAFL usage
- consolidate i18n namespaces
- cleanup obsolete docs after reference search
