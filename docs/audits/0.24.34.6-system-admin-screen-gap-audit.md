# 0.24.34.5 System Admin Screen Gap Audit

Status: `CONFIRMED_AUDIT`
Target version: `0.24.34.6`
Baseline: `0.24.34.5` policy-docs commit `57411acb`
Purpose: record route-level gaps before `0.24.35 — Company-wide Export Execution` starts.

## 1. Audit rule

This audit is not a completion claim. It records what must still be implemented, verified, or explicitly deferred.

Every system-admin screen must be judged by:

- actual DB/API connection versus mock/design-only state,
- production/customer data safety,
- Korean operator-facing copy,
- WAFL common component usage,
- desktop/mobile/iPad usability,
- route-level permission boundary,
- evidence required before Product Verified,
- whether the work belongs to `0.24.34.5` completion, `0.24.35` Export, or a later production sprint.

## 3. `/system` operations dashboard

Current problem:

- 확인 필요, 가입 검토, Trial, 저장공간, 요금제 분포, 고객사 운영현황이 분리되어 있다.
- 고객사 목록이 길게 내려가며 운영자가 빠르게 판단하기 어렵다.
- tablet landscape에서 card/list density가 낮고, 중요한 운영 queue가 밀린다.
- system-admin 운영 화면과 customer workspace dashboard 정책이 일부 섞여 보일 수 있다.

Required product direction:

- one compact operations dashboard,
- top summary cards: 확인 필요, 가입 검토 대기, Trial 종료 임박, 저장공간 위험,
- compact plan/status distribution,
- customer operations table with 10-20 rows per page,
- search, status filter, plan filter, attention-only filter, sort, pagination or show-more,
- row click opens company detail,
- desktop table, tablet hybrid, mobile cards,
- raw enum/internal English hidden.

Status: `TODO`
Priority: `HIGH`
Target: `0.24.34.5 continuation E` before `0.24.35`.

## 4. `/system/signup-applications`

Current problem:

- signup review flow still risks too many visible actions for real operation,
- `검토 시작` should not remain a primary operator CTA if review can be inferred from opening detail,
- provisioning/readiness/internal-state vocabulary must be fully audited in rendered UI,
- 보완 요청 -> applicant return -> resubmit -> approve flow needs live/browser evidence,
- same Google returning-applicant round trip remains not fully verified.

Required product direction:

- main actions for submitted/reviewing application: `승인`, `보완 요청`, `반려`, `더보기`,
- review start is automatic on first detail open or moved out of primary CTA,
- more menu only: internal memo, audit history, special recovery, cancel handling,
- detail drawer/modal sections: company info, applicant info, certificate, policy agreement, history, internal memo, approve/revision/reject,
- rendered status mapping in Korean:
  - `SUBMITTED` -> `검토 대기`
  - `UNDER_REVIEW` -> `검토 중`
  - `NEEDS_REVISION` -> `보완 요청`
  - `APPROVED` -> `승인 완료`
  - `REJECTED` -> `반려`
  - `CANCELLED` -> `신청 취소`

Status: `TODO`
Priority: `HIGH`
Target: `0.24.34.5 continuation D/E`.

## 5. `/system/companies`

Current decision:

- do not delete this route.
- `/system/signup-applications` is pre-approval review.
- `/system/companies` is approved customer operations management.

Required product direction:

- compact customer list,
- company status, plan, Trial end date, user count, storage usage, payment status, latest activity, attention badge,
- company detail drawer/page,
- approve flow links to this route via `고객사 열기`,
- route/deep link/audit-log compatibility preserved,
- duplicate dashboard-only cards removed or moved to `/system`.

Status: `TODO`
Priority: `HIGH`
Target: `0.24.34.5 continuation E`.

## 6. `/system/billing`

Current production boundary:

- PG/provider and real payment execution remain deferred until provider/legal/contract decisions.
- the screen must not imply production billing is live if it is not.

Required product direction before launch:

- show payment readiness accurately,
- distinguish Trial, active paid, suspended, terminated, and payment pending,
- hide internal PG/webhook terminology from operators unless in diagnostic mode,
- link customer billing status to `/system/companies`,
- mark real PG mutation as `DEFERRED` until provider integration sprint.

Status: `DEFERRED_PARTIAL`
Priority: `MEDIUM`
Target: post-0.24.35 billing production sprint unless required by QA.
