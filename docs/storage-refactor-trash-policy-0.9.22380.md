# 저장소 관리 리팩토링 1차 — 휴지통 정책 상수화

## 버전
0.9.22380

## 목적
저장소 관리 리팩토링을 시작하면서 작업지시서 삭제 묶음 판단에 쓰이던 문장 기반 비교를 한 곳으로 모았다.

## 변경 기준
- `delete_reason` 문장을 각 파일에 직접 하드코딩하지 않는다.
- 작업지시서 삭제로 함께 휴지통 이동한 첨부파일 판단은 `ADMIN_FILE_TRASH_REASONS.workorderBundle`을 기준으로 한다.
- 문구 비교가 필요한 곳은 `isWorkOrderBundleTrashReason`을 사용한다.
- DB schema 변경 없이 현재 구조에서 가능한 최소 리팩토링으로 제한했다.

## 후속 검토
현재 DB에는 별도 reason code 컬럼이 없어서 `delete_reason` 값을 정책 상수로 중앙화하는 수준으로 정리했다. 다음 구조 개선에서는 아래 중 하나를 검토한다.

1. `delete_reason_code` 컬럼 추가
2. `delete_reason`은 표시 문구로만 사용하고 판단은 code 컬럼으로 분리
3. 기존 데이터 마이그레이션 정책 수립

## 확인 항목
- 작업지시서 삭제 시 함께 삭제된 첨부파일만 묶음 복원/묶음 purge 대상이 되는지 확인
- 먼저 개별 삭제된 첨부파일은 기존 휴지통 흐름에 남는지 확인
- 시스템관리자 실제 삭제 후보에서 작업지시서 대표 row와 파일 후보가 분리되는지 확인
