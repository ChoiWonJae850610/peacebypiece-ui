# 제품화 전 QA 오류 수정 1차 (0.11.52)

## 목적

0.11.51에서 시작한 QA 오류 수정 라인의 첫 번째 점검 버전이다. 이번 입력에는 build log, runtime error, 화면 캡처가 첨부되지 않았으므로 실제 기능 코드는 수정하지 않고, 다음 QA 실행 시 바로 확인할 수 있는 오류 분류 기준과 기록 양식을 구체화한다.

## 기준 버전

- 현재 기준 원본: 0.11.51
- 결과 버전: 0.11.52

## 이번 버전에서 실제 수정하지 않은 이유

- 새 build log가 첨부되지 않았다.
- runtime error stack이 첨부되지 않았다.
- 화면 캡처 기반 UI 오류가 첨부되지 않았다.
- repo-state는 clean 상태이므로 임의 추정으로 기능 코드를 수정하지 않는다.

## QA 오류 분류 기준

### P0: 즉시 차단

- `npm run build` 실패
- 작업지시서 목록/상세 DB 로딩 실패
- 로그인/초대/승인 흐름 전체 차단
- 첨부 업로드/다운로드/R2 삭제 실패
- 저장소 휴지통 복원/삭제 요청 실패
- 시스템관리자 실제 삭제 처리 실패

### P1: 제품화 전 필수 수정

- 모바일/태블릿에서 주요 버튼 접근 불가
- table/list overflow로 데이터 확인 불가
- modal footer 버튼이 화면 밖으로 밀림
- i18n 전환 시 고객 노출 문구가 깨짐
- 통계 차트가 모바일에서 잘리거나 tooltip 접근이 어려움

### P2: 출시 전 정리 권장

- 안내 문구가 길거나 중복됨
- card/section 간격이 화면마다 다름
- empty/loading/error 표현이 화면마다 미세하게 다름
- 문서와 실제 화면 명칭이 다름

## QA 기록 양식

```text
버전:
화면:
기기/폭:
재현 단계:
현재 결과:
기대 결과:
첨부 자료:
판단 우선순위: P0 / P1 / P2
```

## 다음 패치에서 우선 확인할 입력 자료

1. build log가 있으면 TypeScript/build 오류를 최우선 수정한다.
2. runtime error가 있으면 stack 기준으로 원인을 먼저 좁힌다.
3. 화면 캡처가 있으면 해당 화면만 최소 수정한다.
4. 기능이 정상인데 문구/간격 문제라면 제품화 polish 라인으로 분리한다.

## 수정 금지 기준

- 오류 자료 없이 작업지시서 저장/상태변경 로직을 임의 수정하지 않는다.
- 첨부/R2 upload/download/delete/purge 흐름을 임의 수정하지 않는다.
- DB schema는 명시적 오류 또는 설계 변경이 없으면 수정하지 않는다.
- package.json/package-lock.json은 필요성이 명확할 때만 수정한다.

## 0.11.53 권장 방향

사용자 로컬 build log, runtime error, 화면 캡처 중 하나 이상을 기준으로 실제 오류를 작은 범위로 수정한다.
