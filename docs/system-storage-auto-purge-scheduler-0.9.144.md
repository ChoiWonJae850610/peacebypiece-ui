# 0.9.144 자동 purge 스케줄러 도입 검토

## 목적

`/system/storage-usage`에서 시스템관리자가 수동으로 R2 purge를 실행할 수 있는 상태가 되었으므로, 다음 단계로 자동 purge 스케줄러를 도입할지 검토한다. 이번 버전에서는 실제 자동 실행 기능을 만들지 않고, 운영 방식과 책임 경계를 문서화한다.

## 현재 확정된 운영 기준

- 고객사별 삭제 보관 기간 설정은 사용하지 않는다.
- 모든 고객사는 공통 30일 휴지통 보관 정책을 따른다.
- 파일 삭제 후 30일이 지나면 R2 실제 삭제 후보가 된다.
- 시스템관리자는 `/system/storage-usage`에서 후보 목록을 확인하고 수동 purge를 실행할 수 있다.
- purge 대상은 원본 `storage_key`와 썸네일 `thumbnail_key`를 함께 포함한다.
- R2 직접 SDK 삭제 방식은 사용하지 않고 Worker 기반 삭제 흐름을 유지한다.

## 선택지 비교

| 방식 | 장점 | 단점 | 현재 판단 |
| --- | --- | --- | --- |
| 시스템관리자 수동 실행 유지 | 가장 안전하고 예측 가능하다. 실수나 대량 삭제를 눈으로 확인할 수 있다. | 운영자가 주기적으로 확인해야 한다. | 초기 운영 기본값으로 적합하다. |
| Vercel Cron | Next.js/Vercel API에서 DB 후보 조회와 실행 흐름을 관리하기 쉽다. | cron 설정, 인증, 실행 실패/중복 실행 방지 설계가 필요하다. | 자동화 후보 1순위이나 아직 이르다. |
| Cloudflare Scheduled Trigger | R2와 같은 Cloudflare 영역에서 스케줄 실행을 관리할 수 있다. | Neon DB 후보 조회까지 Worker가 담당하면 DB 접근 경계가 복잡해진다. | R2 삭제만 Worker가 담당하는 현재 구조와는 부분적으로만 적합하다. |

## 권장 구조

자동 purge를 도입하더라도 DB 후보 조회와 상태 갱신은 Vercel API가 담당하고, R2 객체 삭제는 Cloudflare Worker가 담당하는 구조를 유지한다.

```text
Vercel Cron 또는 수동 시스템관리자 액션
→ Next.js API에서 purge 후보 조회
→ Worker 삭제 URL/API 호출
→ 원본 storage_key + thumbnail_key 삭제
→ DB purge 상태 갱신
```

이 구조를 유지하면 UI, 수동 실행, 향후 자동 실행이 같은 server-side purge runner를 공유할 수 있다.

## 자동 purge 도입 전 필요한 조건

1. 수동 purge가 여러 번 안정적으로 검증되어야 한다.
2. 실패 항목 재시도와 실패 사유 표시가 충분히 정리되어야 한다.
3. 중복 실행 방지 기준이 있어야 한다.
4. 대량 삭제 시 batch size 제한이 있어야 한다.
5. 삭제 전 후보 재검증이 있어야 한다.
6. 자동 실행 결과를 시스템관리자 화면에서 확인할 수 있어야 한다.
7. 긴급 중지 또는 자동 purge OFF 정책이 있어야 한다.

## 초기 운영 권장안

초기에는 자동 purge를 켜지 않는다.

```text
초기 운영:
- 30일 경과 후보 자동 산정
- 시스템관리자 화면에서 후보 확인
- 시스템관리자가 수동 purge 실행
- 실패 항목은 화면에서 확인 후 재시도
```

자동화는 후보 목록과 수동 실행이 충분히 안정화된 뒤 도입한다.

## 자동 purge 도입 시 권장 단계

### 1단계 — 수동 실행 유지

- 현재처럼 `/system/storage-usage`에서 시스템관리자가 실행한다.
- 실제 고객 데이터 기준으로 후보 수, 실패율, 삭제 시간, Worker 응답을 확인한다.

### 2단계 — dry-run API 추가

- 실제 삭제 없이 후보 수와 예상 용량만 계산한다.
- cron 도입 전 운영자가 결과를 확인할 수 있게 한다.

### 3단계 — Vercel Cron 검토

- 하루 1회 또는 주 1회 실행을 검토한다.
- 환경 변수 기반 secret으로 cron API를 보호한다.
- batch size를 제한한다.
- 실행 결과를 DB 또는 시스템관리자 화면에 남긴다.

### 4단계 — 자동 purge ON/OFF 정책

- 기본값은 OFF로 둔다.
- 운영자가 시스템 설정에서 켤 수 있게 한다.
- 실패율이 높거나 Worker 오류가 발생하면 자동으로 중단하는 기준을 둔다.

## 이번 버전에서 하지 않는 일

- Vercel Cron 설정 추가 없음
- Cloudflare Scheduled Trigger 추가 없음
- 자동 purge API 추가 없음
- DB schema 변경 없음
- R2 직접 SDK 삭제 없음
- package.json / package-lock.json 변경 없음
