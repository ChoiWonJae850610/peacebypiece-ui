# Admin Files Stability — 0.9.136

## 목적

0.9.77 복구 라인 기준의 `/admin/files` 본 기능을 유지하면서 파일 목록, 휴지통, 복구, 영구삭제 요청 흐름의 UI 안정성을 보완한다.

이번 버전은 R2 실제 삭제 방식이나 DB schema를 변경하지 않는다. 작업지시서 첨부 삭제와 관리자 파일관리 삭제 흐름을 구분하고, 관리자 파일관리에서 중복 액션 요청이 발생하지 않도록 방지하는 것을 우선한다.

## 현재 구조

- `/admin/files`는 DB snapshot API를 통해 첨부 목록과 휴지통 목록을 조회한다.
- 일반 첨부 삭제는 `/api/workorders/attachments/delete`를 호출해 attachments soft delete와 `attachment_trash_items` 기록을 사용한다.
- 휴지통 복구는 `/api/admin/files/trash/restore`를 호출한다.
- 영구삭제 요청은 `/api/admin/files/trash/purge`를 호출해 `purge_requested` 상태로 표시한다.
- R2 실제 삭제는 별도 purge worker 흐름에서 처리한다.

## 0.9.136 보완 내용

- `/admin/files` 페이지에 `pendingFileAction` 상태를 추가했다.
- 휴지통 이동, 복구, 영구삭제 요청 중에는 같은 화면의 파일 액션 버튼을 비활성화한다.
- 저장소 목록/휴지통 테이블의 전체 선택 버튼도 액션 진행 중에는 비활성화한다.
- 액션 중 버튼 문구는 `처리 중`으로 표시한다.
- 액션 성공 후 기존처럼 선택 상태를 초기화하고 snapshot을 새로고침한다.

## 변경하지 않은 것

- R2 실제 삭제 방식 변경 없음
- Worker 기반 삭제 흐름 변경 없음
- 첨부 업로드/다운로드/미리보기 변경 없음
- DB schema 변경 없음
- API 응답 포맷 변경 없음
- 파일 목록/휴지통 테이블 구조 변경 없음

## QA 체크리스트

### 1. 화면 진입

- `/admin/files` 진입 시 runtime error가 없어야 한다.
- 첨부파일 목록과 휴지통 탭이 표시되어야 한다.
- 저장소 요약 카드가 기존처럼 표시되어야 한다.

### 2. 첨부 삭제 요청

1. 첨부파일 목록에서 1개 이상 선택한다.
2. 삭제 버튼을 빠르게 여러 번 클릭한다.
3. 네트워크 요청이 중복으로 반복되지 않는지 확인한다.
4. 처리 중에는 삭제/전체 선택 버튼이 비활성화되는지 확인한다.
5. 성공 후 목록이 새로고침되고 선택 상태가 초기화되는지 확인한다.

### 3. 휴지통 복구

1. 휴지통 탭으로 이동한다.
2. 휴지통 항목을 1개 이상 선택한다.
3. 복구 버튼을 빠르게 여러 번 클릭한다.
4. 처리 중에는 복구/영구 삭제/전체 선택 버튼이 비활성화되는지 확인한다.
5. 성공 후 휴지통 목록에서 해당 항목이 사라지고 첨부파일 목록에 복구되는지 확인한다.

### 4. 영구삭제 요청

1. 휴지통 항목을 선택한다.
2. 영구 삭제 버튼을 클릭한다.
3. 실제 R2 삭제가 즉시 일어나지 않고, 영구삭제 요청 상태로 전환되는지 확인한다.
4. R2 실제 삭제는 purge worker 대상이므로 이 화면에서 직접 SDK 삭제가 발생하면 안 된다.

### 5. 회귀 확인

- 작업지시서 첨부 업로드 정상
- 작업지시서 첨부 다운로드 정상
- 작업지시서 첨부 삭제 정상
- 메모 저장 후 상태 변경/새로고침 시 유지 정상
- 첨부 카드 썸네일 표시 정상
