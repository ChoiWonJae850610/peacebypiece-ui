# 0.9.22403 작업지시서 휴지통 정책 보정

## 목표

작업지시서 삭제/복원/삭제 요청 흐름에서 문서/디자인/메모 묶음 항목을 `delete_reason` 문장에만 의존하지 않고 구조화된 삭제 메타데이터 기준으로 우선 판정한다.

## 반영 내용

- 작업지시서 묶음 삭제 판정 SQL을 `trashPolicy.ts` 공통 함수로 분리했다.
- `delete_source = workorder_bundle`, `delete_scope = bundle`, `delete_parent_type = workorder`, `delete_parent_id` 기준을 우선 사용한다.
- 기존 데이터 호환을 위해 `delete_reason = 작업지시서 삭제로 함께 휴지통 이동` fallback은 유지한다.
- 작업지시서 복원 시 메모 복원 대상도 삭제 메타데이터 기준으로 찾도록 보정했다.
- 작업지시서 삭제 요청 시 이미 선삭제된 독립 메모가 묶음 삭제 요청으로 잘못 흡수되지 않도록, 메모 대상 조건을 작업지시서 묶음 메타데이터 기준으로 제한했다.
- 작업지시서 삭제 요청으로 넘어가는 첨부/휴지통 row에는 누락된 묶음 삭제 메타데이터를 보강한다.
- 시스템관리자 실제 삭제 후보 판정도 동일한 공통 SQL predicate를 사용하도록 중복 문자열 비교를 줄였다.

## DB 변경

없음.

0.9.22399에서 추가한 컬럼을 사용하지만, 이 버전은 추가 DDL을 포함하지 않는다.

## 후속 작업

- 0.9.22404에서 시스템관리자 실제 삭제 후보 화면과 purge 흐름을 기준으로 다시 점검한다.
- legacy `delete_reason` fallback 제거 시점은 저장소관리 안정화 이후 별도 버전에서 결정한다.
