## Objective

Verify and fix the real dev/test workorder flow, especially legacy/existing workorder selection, factory save, generated PDF attachment refresh, and error-state behavior.

## Known owner-reported defects

- Existing workorder selection can hang or enter a loading state.
- New workorder creation may render while existing record detail path fails.
- Factory list loads, but factory save may fail with an English 404-style message.
- After save failure the workorder can return to a loading state.
- Generated PDF viewer previously failed or showed unreadable PDF.

## Required live-dev/test scenarios

Use actual dev/test session and DB/API paths. Do not rely only on mocked route fixtures.

1. `/workspace/workorders` initial load.
2. Existing workorder select.
3. Middle detail panel renders.
4. Right panel renders.
5. Factory list loads.
6. Factory select/save.
7. Save succeeds and persists after refresh.
8. Save failure fixture shows Korean error and does not break the screen.
9. Generated incomplete PDF appears in generated documents/attachments.
10. PDF viewer opens.
11. Different existing workorder can be selected after an error.
12. No spinner persists beyond the defined timeout.

## Required UI behavior

- List remains usable if detail fails.
- Detail error is scoped to detail panel.
- Right-panel error is scoped to right panel.
- Factory save pending/error is scoped to factory instruction panel.
- PDF failure is scoped to generated-document/PDF area.
- All user-facing errors are Korean.
- Raw English error text and internal status codes are not shown to customers.
