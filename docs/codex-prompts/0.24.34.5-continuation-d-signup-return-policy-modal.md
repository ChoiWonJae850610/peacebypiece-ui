## Objective

Fix the signup return flow and unify policy viewing/consent evidence.

## Owner-reported defect

After submitting a new company signup application, closing Chrome, and signing in again with the same Google email, the user is shown the business-information form again. The correct behavior is to show the existing application status, such as 검토 대기.

## Required applicant state flow

Same Google account login must resolve:

- no application -> new signup form,
- `DRAFT` -> continue draft,
- `SUBMITTED` -> 검토 대기 screen,
- `UNDER_REVIEW` -> 검토 중 screen,
- `NEEDS_REVISION` -> 보완 요청 screen and edit/resubmit flow,
- `APPROVED` -> workspace,
- `REJECTED` -> 반려 안내,
- `CANCELLED` -> 취소 안내 and policy-defined reapply option.

## Policy viewer

Public signup policy links must use the same canonical policy viewer/content used by system/customer settings.

Required:

- shared policy viewer component,
- same policy ID,
- same version,
- effective date,
- full content,
- loading/error/retry,
- canonical modal/sheet,
- ESC/focus trap/scroll lock,
- no hardcoded public-signup summary copy as the legal source.

Consent evidence should include:

- policy type,
- policy version,
- document hash,
- agreedAt,
- applicant/member/company where available,
- agreement source,
- required/optional,
- locale.

If schema changes are needed for hash persistence, document the additive migration requirement and stop before destructive changes.
