# PeaceByPiece 작업지시서 핵심 Flow QA 0.9.129

## 기준

- 기준 버전: `0.9.128`
- 결과 버전: `0.9.129`
- 기준 라인: `0.9.77` 안정 UI 복구 라인
- 목적: 작업지시서의 핵심 상태 전환과 데이터 유지 여부를 확인하는 회귀 테스트 기준을 고정한다.
- 작업 범위: 기능 코드 변경 없음. APP_VERSION 갱신과 QA 문서 추가/갱신만 수행한다.

## 이번 버전에서 지킨 제한

1. `0.9.108` 이후 read-only/skeleton route 변경은 가져오지 않았다.
2. DB schema는 변경하지 않았다.
3. `package.json`, `package-lock.json`, `.env.local`은 변경하지 않았다.
4. 작업지시서 UI, actionFlow, workflowPolicy, repository 코드는 변경하지 않았다.
5. R2 직접 SDK 업로드/삭제 fallback은 재도입하지 않았다.
6. 첨부와 메모의 정상 동작 흐름은 변경하지 않았다.

## 핵심 확인 관점

작업지시서 회귀 테스트는 단순히 버튼이 눌리는지만 보는 것이 아니라, 상태 변경 뒤에도 DB와 화면이 같은 상태를 유지하는지를 확인해야 한다.

확인 축은 다음 네 가지다.

1. **상태 전환**: 작성중, 검토요청, 검토완료, 발주요청, 생산/검수, 완료가 기대 순서대로 이동하는지 확인한다.
2. **데이터 유지**: 제목, 담당자, 납기일, 수량, 발주정보, 생산구성, 첨부, 대표 이미지, 메모가 상태 변경 후 사라지지 않는지 확인한다.
3. **저장 위치**: 화면 로컬 상태에만 남는 것이 아니라 DB 테이블에도 저장되는지 확인한다.
4. **새로고침 회귀**: 상태 변경 후 새로고침하거나 목록에서 다시 열어도 같은 값이 유지되는지 확인한다.

## 테스트 전 준비

### 실행

```bash
npm run build
npm run dev
```

### 기본 화면

- `/` 작업지시서 메인
- `/worker` 작업자 화면
- `/admin` 고객관리자 콘솔
- `/admin/files` 파일관리
- `/admin/partners` 거래처/공장관리

### DB 확인 권장 테이블

프로젝트 DB 명칭은 실제 schema 기준으로 확인한다. 현재 복구 라인에서 주로 확인할 후보는 다음이다.

- `workorders` 또는 작업지시서 본문 테이블
- `attachments`
- `attachment_trash_items`
- `memos`
- 발주정보 관련 테이블
- 생산구성/공장 발주 관련 테이블
- 거래처/공장 관련 테이블

테이블명이 실제 schema와 다르면, UI 기능 기준으로 대응되는 테이블을 확인한다.

## Flow 1. 작성중 기본 저장

### UI 절차

1. 신규 작업지시서를 생성한다.
2. 제목을 변경한다.
3. 담당자를 변경한다.
4. 납기일, 수량, 카테고리, 시즌 등 기본 정보를 입력한다.
5. 발주정보 또는 생산구성 정보를 저장한다.
6. 새로고침 후 같은 작업지시서를 다시 연다.

### 기대 결과

- 제목과 담당자가 유지된다.
- 기본 정보가 유지된다.
- 발주정보와 생산구성 정보가 유지된다.
- 화면의 상태는 `작성중` 기준으로 유지된다.
- DB의 작업지시서 row가 갱신된다.

### 실패로 판단할 증상

- 새로고침 후 제목이 이전 값으로 돌아간다.
- 담당자가 바뀌었는데 상태가 의도치 않게 변경된다.
- 발주정보/생산구성 저장 버튼은 성공처럼 보이지만 DB에는 반영되지 않는다.

## Flow 2. 메모 저장과 상태 변경 후 유지

### UI 절차

1. 작성중 작업지시서에서 메모를 작성한다.
2. DB의 `memos` 계열 테이블에 row가 생성되는지 확인한다.
3. 검토요청 또는 검토완료로 상태를 변경한다.
4. 작업지시서를 새로고침한다.
5. 같은 메모가 화면에 남아 있는지 확인한다.

### 기대 결과

- 메모 저장 시 `company_id` not-null 오류가 발생하지 않는다.
- `parent_id` FK 오류가 발생하지 않는다.
- 상태 변경 후에도 메모가 사라지지 않는다.
- 화면 로컬 상태가 아니라 DB에 저장된 메모가 다시 hydrate된다.

### 실패로 판단할 증상

- 메모는 화면에 보이지만 DB row가 없다.
- 상태 변경 후 메모가 사라진다.
- 콘솔에 `memos.company_id` 또는 `memos_parent_id_fkey` 오류가 다시 나온다.

## Flow 3. 디자인 첨부 업로드, 다운로드, 삭제

### UI 절차

1. 디자인 첨부 이미지를 1개 업로드한다.
2. R2에 파일이 생성되는지 확인한다.
3. DB의 attachment metadata가 생성되는지 확인한다.
4. 화면에서 미리보기와 다운로드를 확인한다.
5. 같은 첨부를 삭제한다.
6. 삭제 후 화면에서 사라지는지 확인한다.
7. `/admin/files`에서 휴지통 또는 삭제 후보가 의도대로 표시되는지 확인한다.

### 기대 결과

- 업로드는 Worker 경로를 사용한다.
- `/api/workorders/attachments/upload/direct` 서버 fallback이 호출되지 않는다.
- `SSL EPROTO` handshake 오류가 업로드 성공 경로에 남지 않는다.
- 삭제는 DB metadata 상태 변경과 관리자 파일관리 흐름을 따른다.
- R2 물리 삭제는 작업지시서 일반 삭제와 섞이지 않는다.

### 실패로 판단할 증상

- R2에는 올라가지만 `/upload/direct 500`이 다시 발생한다.
- 삭제 후 화면에는 사라졌지만 DB 상태가 변경되지 않는다.
- 삭제 후 새로고침하면 첨부가 다시 나타난다.

## Flow 4. 대표 이미지 회귀 확인

### UI 절차

1. 디자인 첨부를 2개 업로드한다.
2. 하나를 대표 이미지로 지정한다.
3. DB에서 대표 이미지가 1개만 `true`인지 확인한다.
4. 새로고침 후 대표 표시가 유지되는지 확인한다.
5. 대표 이미지를 삭제한다.

### 기대 결과

- 대표 이미지는 같은 작업지시서의 디자인 첨부 기준 1개만 유지된다.
- 대표 이미지 지정 후 새로고침해도 유지된다.
- 대표 이미지 삭제 시 자동 승계가 아직 구현되지 않았더라도, 삭제된 첨부가 대표 이미지로 남지는 않아야 한다.

### 현재 보류 중인 미세 로직

아래는 당장 실패로 보지 않고 0.9.130~0.9.131에서 묶어서 처리한다.

- 디자인 첨부가 없는 상태에서 최초 업로드 시 자동 대표 지정
- 대표 이미지 삭제 시 남은 디자인 첨부 자동 대표 승계
- 대표 이미지가 없는 상태에서 안내 문구 표시
- 첨부 삭제 후 preview/selection 상태 초기화

## Flow 5. 검토요청

### UI 절차

1. 디자이너 담당 작업지시서에서 작성중 상태를 준비한다.
2. 필수 입력값을 채운다.
3. 검토요청을 실행한다.
4. 관리자 화면 또는 권한 전환 후 같은 작업지시서를 확인한다.

### 기대 결과

- 상태가 `review_requested`로 변경된다.
- 작성중에서 입력한 기본정보, 발주정보, 생산구성, 첨부, 메모가 유지된다.
- 검토요청 이후 일반 편집 잠금 정책이 적용된다.
- 관리자는 검토완료 또는 반려 action을 볼 수 있다.

### 실패로 판단할 증상

- 검토요청 후 데이터 일부가 사라진다.
- 디자이너가 잠겨야 할 값을 계속 수정할 수 있다.
- 관리자가 검토 action을 볼 수 없다.

## Flow 6. 검토완료와 취소

### UI 절차

1. 관리자로 검토요청 작업지시서를 연다.
2. 검토완료를 실행한다.
3. 발주요청 가능 상태인지 확인한다.
4. 검토완료 취소를 실행한다.

### 기대 결과

- 상태가 `review_completed`로 변경된다.
- 검토완료 후 데이터가 유지된다.
- 검토완료 취소 시 담당자 역할 정책에 따라 `draft` 또는 `review_requested`로 돌아간다.
- 메모 row는 상태 변경으로 삭제되지 않는다.

### 실패로 판단할 증상

- 검토완료 후 메모/첨부/발주정보가 사라진다.
- 취소 후 상태가 정책과 다르게 돌아간다.

## Flow 7. 발주요청과 생산/검수

### UI 절차

1. 검토완료 상태에서 발주요청을 실행한다.
2. 공장/납기일/수량 등 필수값 validation을 확인한다.
3. 발주요청 후 생산/검수 단계로 이동하는지 확인한다.
4. 검수 상태값을 변경한다.

### 기대 결과

- 상태가 `inspection`으로 변경된다.
- orderEntries의 inspectionStatus가 발주요청 기준 상태로 갱신된다.
- 발주정보 snapshot이 유지된다.
- 기존 첨부/메모는 유지된다.

### 실패로 판단할 증상

- 발주요청 후 orderEntries가 비거나 inspectionStatus가 깨진다.
- 발주정보가 저장되지 않고 상태만 바뀐다.
- 메모/첨부가 사라진다.

## Flow 8. 완료와 재검수요청

### UI 절차

1. 생산/검수 단계의 검수 항목을 완료 처리한다.
2. 작업지시서를 완료 상태로 이동한다.
3. 완료 상태에서 재검수요청 가능 여부를 확인한다.
4. 재검수요청 후 다시 생산/검수 단계로 돌아가는지 확인한다.

### 기대 결과

- 상태가 `completed`로 변경된다.
- 완료 후 작업지시서 핵심 데이터가 유지된다.
- 재검수요청 시 상태가 `inspection`으로 돌아가고, 검수 상태가 재검수 기준으로 초기화된다.

### 실패로 판단할 증상

- 완료 후 화면 진입 시 runtime error가 발생한다.
- 재검수요청이 재고수정 모달 등 다른 modal을 잘못 연다.
- 완료 후 첨부/메모가 사라진다.

## 역할별 체크

### 관리자

- 관리자 생성 작업지시서는 직접 검토완료 가능해야 한다.
- 디자이너가 검토요청한 작업지시서는 반려 또는 검토완료 가능해야 한다.
- 검토완료 취소는 담당자 역할 정책을 따른다.
- 발주요청은 검토완료 이후 가능해야 한다.

### 디자이너

- 본인이 담당한 작성중 작업지시서를 수정할 수 있어야 한다.
- 검토요청 후 잠금 정책이 적용되어야 한다.
- 반려 상태에서는 다시 수정하고 검토요청할 수 있어야 한다.

### 검수담당자

- 생산/검수 단계에서 필요한 검수 정보를 확인할 수 있어야 한다.
- 권한 밖의 기본정보/발주정보 수정은 제한되어야 한다.

## 최소 회귀 테스트 세트

시간이 부족하면 아래만 우선 수행한다.

1. 신규 작업지시서 생성 → 제목 변경 → 새로고침 유지
2. 메모 작성 → 검토완료 → 새로고침 후 메모 유지
3. 이미지 첨부 업로드 → R2/DB 확인 → 다운로드 확인
4. 첨부 삭제 → 화면/DB 상태 확인
5. 검토요청 → 검토완료 → 발주요청 → 생산/검수 진입 확인
6. `npm run build` 통과 확인
