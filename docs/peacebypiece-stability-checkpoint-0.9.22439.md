# 0.9.22439 — 작업지시서/저장소 안정 버전 확정

## 목적

작업지시서 lazy load, 고객관리자 저장소/휴지통, 시스템관리자 실제 삭제 후보 흐름을 다음 관리자 화면 구조 개편 전 기준점으로 확정한다.

이번 버전은 기능 추가보다 안정 체크포인트 문서화를 우선한다. 정상 동작 중인 작업지시서 저장, 첨부, 메모, 삭제, 복원, purge 로직은 변경하지 않는다.

## 확정 범위

### 1. 작업지시서 lazy load

현재 기준 구조는 다음과 같다.

1. `/worker` 첫 진입 시 전체 상세 hydrate 대신 `/api/workorders/summary`를 먼저 호출한다.
2. 목록은 summary 응답으로 표시한다.
3. 사용자가 작업지시서를 선택했을 때만 `/api/workorders/{workOrderId}` 상세 조회를 수행한다.
4. 상태 변경과 검수 완료는 전체 저장이 아니라 상태 patch 중심으로 처리한다.
5. 상태 patch 응답은 전체 작업지시서, 첨부, 메모 thread를 다시 싣지 않는다.
6. `/api/partners/workorder-options`는 hook cache 기준으로 선택마다 반복 호출되지 않아야 한다.
7. 리오더 생성은 새 리오더 1건 생성 응답을 목록에 병합하는 흐름을 기준으로 한다.

### 2. 고객관리자 저장소/휴지통

현재 기준 정책은 다음과 같다.

1. `/admin/files`는 복원 가능한 휴지통 항목 중심으로 표시한다.
2. `purge_requested` 항목은 고객관리자 휴지통 기본 목록에서 숨긴다.
3. 고객관리자의 삭제 요청은 실제 R2 삭제가 아니라 시스템관리자 처리 대기 상태로 본다.
4. 작업지시서 삭제 시 함께 삭제된 문서, 디자인, 메모는 작업지시서 bundle metadata를 기준으로 묶는다.
5. 작업지시서 복원 시 작업지시서 삭제 당시 살아 있던 묶음 항목만 같이 복원한다.
6. 작업지시서 삭제 전에 이미 단독 삭제된 파일/메모는 자기 삭제일시와 삭제 상태를 유지한다.
7. 고객-facing 문구에서는 `연결 첨부` 같은 표현을 쓰지 않고 `문서 n개, 디자인 n개, 메모 n개`처럼 역할 기반으로 표시한다.
8. 삭제일시는 KST 표시 기준을 유지한다.

### 3. 시스템관리자 실제 삭제 후보

현재 기준 정책은 다음과 같다.

1. `/system/storage-usage`는 고객관리자가 삭제 요청한 항목을 실제 삭제 후보로 표시한다.
2. 작업지시서 후보는 작업지시서 1건과 함께 처리해야 하는 문서, 디자인, 메모 개수를 표시한다.
3. R2 object 삭제가 성공한 뒤 DB를 `purged` 상태로 갱신한다.
4. R2 object가 이미 없거나 not found 계열이면 삭제 성공으로 처리한다.
5. Worker 인증 실패, 권한 실패, 네트워크 실패는 실패 후보로 남겨 재시도 대상으로 유지한다.
6. 작업지시서 실제 삭제 시 memo는 작업지시서 bundle metadata가 있는 항목만 purged 처리한다.

## 회귀 테스트 기준

### 작업지시서

1. `/worker` 첫 진입 시 목록이 summary 기준으로 표시되는지 확인한다.
2. 작업지시서 선택 시 선택한 1건만 상세 조회되는지 확인한다.
3. 선택 해제 후 재선택 시 화면이 깨지지 않는지 확인한다.
4. 검수 완료 후 저장중 메시지가 무한 지속되지 않는지 확인한다.
5. 리오더 생성 후 pending 상태가 오래 유지되지 않는지 확인한다.
6. 새 리오더가 새로고침 없이 목록에 반영되는지 확인한다.

### 고객관리자 저장소/휴지통

1. `/admin/files` 저장소 화면에 진입한다.
2. 작업지시서 삭제 후 휴지통에 표시되는지 확인한다.
3. 작업지시서 상세 모달에서 진행 단계 위치가 실제 상태와 맞는지 확인한다.
4. 작업지시서 복원 시 묶음 삭제된 문서, 디자인, 메모만 같이 복원되는지 확인한다.
5. 작업지시서 삭제 전에 단독 삭제한 파일/메모가 작업지시서 복원으로 살아나지 않는지 확인한다.
6. 문서, 디자인, 메모 단독 삭제/복원/삭제 요청이 각각 분리되어 동작하는지 확인한다.
7. 삭제일시가 작업지시서 묶음 삭제와 단독 삭제 기준으로 구분되는지 확인한다.

### 시스템관리자 실제 삭제

1. `/system/storage-usage`에 진입한다.
2. 고객관리자가 삭제 요청한 후보가 표시되는지 확인한다.
3. 작업지시서 후보에 문서, 디자인, 메모 개수가 표시되는지 확인한다.
4. 실제 삭제 실행 시 R2 삭제 후 DB가 purged 처리되는지 확인한다.
5. R2 object not found가 성공 처리되는지 확인한다.
6. Worker 실패/네트워크 실패 상황은 아직 수동 실패 케이스로 남긴다.

## 이번 버전에서 하지 않는 일

- 관리자 IA 화면 구조 변경
- 고객관리자 좌측 패널 제거
- 히스토리 메뉴 제거
- 시스템관리자 감사 로그 화면 추가
- 환경설정 화면 개편
- 요금제/용량 DB 구조 변경
- package.json/package-lock.json 변경
- DB schema 변경

## 완료 판단

아래 조건을 만족하면 0.9.22439를 작업지시서/저장소/휴지통 1차 안정 기준점으로 본다.

1. 작업지시서 첫 진입, 선택, 재선택, 상태 patch, 리오더 생성 흐름이 깨지지 않는다.
2. 고객관리자 휴지통에서 작업지시서 묶음 삭제와 단독 삭제 항목이 섞이지 않는다.
3. 시스템관리자 실제 삭제 후보가 R2 삭제와 DB purge 순서를 지킨다.
4. not found object가 실패로 남지 않는다.
5. 다음 작업을 관리자 IA 구조 개편으로 넘길 수 있다.
