# 요금제 / 저장소 현재 기준

- 기준 앱 버전: `0.24.11`
- 목적: 고객사 구독 상태, 저장공간 한도, 멤버 한도, 업로드 제한 정책을 현재 구현 기준으로 정리한다.
- 제외: PG 연동, 결제수단 저장, 자동 과금, 세금계산서/환불, 운영 DB/R2 직접 조작.

## 현재 데이터 기준

고객사 구독 상태는 고객사 단위로 판단한다. 개인 사용자 단위가 아니라 `company_id` 기준으로 현재 요금제, 구독 상태, 무료체험 기간, 저장공간 한도, 멤버 한도를 조회한다.

현재 핵심 데이터는 다음 경로를 기준으로 한다.

- 구독 상태: `company_subscriptions`
- legacy fallback: `companies`의 plan/status/trial/limit 계열 값
- 회사 파일 사용량: 활성 회사 파일 metadata의 `size_bytes`
- 시스템 저장소 요약: attachment/trash metadata 기반 집계와 명시 snapshot
- 표시 버전 기준: `lib/constants/version.ts`의 `APP_VERSION`

`company_subscriptions` row가 없는 회사는 기존 회사 값으로 fallback할 수 있다. 이 fallback은 과거 데이터 호환을 위한 장치이며, 신규 정책 판단의 기본 기준은 구독 row와 DB-backed usage summary다.

## 요금제와 상태

현재 요금제 코드는 `trial`, `lite`, `flow`, `studio`, `custom` 계열을 기준으로 한다. 실제 결제 연동 전에는 운영자가 상태와 한도를 수동 조정할 수 있는 구조를 우선한다.

구독 상태는 다음 흐름을 기준으로 표시한다.

| 상태 | 기준 동작 |
| --- | --- |
| `trialing`, `active` | 정상 사용 |
| `past_due`, `payment_failed`, `cancel_scheduled` | 경고 또는 확인 필요 상태 |
| `canceled`, `suspended` | 제한 또는 위험 상태 |

결제 실패, 미납, 해지 예정, 해지 완료 상태가 되더라도 기존 업무 데이터는 삭제하지 않는다. 제한은 신규 업로드, 신규 초대, 일부 생성 기능부터 단계적으로 적용한다.

## 저장공간 정책

저장공간 제한은 고객사 단위 파일 업로드 guard로 시작한다.

| 구간 | 기준 | 동작 |
| --- | --- | --- |
| 정상 | 사용률 80% 미만 | 업로드 허용 |
| 경고 | 사용률 80% 이상, 한도 이하 | 업로드 허용 및 경고 |
| 차단 | 예상 사용량이 한도 초과 | 신규 파일 업로드 차단 |

업로드 제한은 UI 안내만으로 처리하지 않는다. 업로드 준비 API와 메타데이터 저장 API 양쪽에서 같은 quota 검사를 수행해 준비 시점과 저장 시점 사이의 race condition을 줄인다.

초과 상태에서도 조회, 다운로드/미리보기, 삭제, 휴지통 정리, 텍스트 기반 업무 입력은 가능한 범위로 유지한다. 저장공간을 줄인 뒤 재시도할 수 있어야 한다.

## 현재 적용 범위

고객사 관리자 환경설정의 `요금제·저장공간` 영역은 현재 요금제, 구독 상태, 저장공간 사용량/한도, 멤버 사용량/한도, 무료체험 종료일, 데이터 출처를 읽기 전용으로 표시한다.

회사 파일 업로드 경로는 저장공간 guard 적용 대상이다.

- 대표 이미지
- 사업자등록증
- 회사 파일 metadata 저장

작업지시서 첨부, 디자인 파일, 일반 문서함, PDF/R2 저장 흐름은 별도 정책과 route별 검증이 필요하다. 직접 목표가 아니면 cleanup 작업에서 변경하지 않는다.

## 권한 범위

- 고객사 관리자: 자기 회사의 요금제, 저장공간, 멤버 사용량, 해결 안내를 볼 수 있다.
- 일반 멤버: 필요한 경우 제한 사유를 볼 수 있지만 요금제 변경 액션은 노출하지 않는다.
- 시스템관리자: 고객사별 저장공간 상태를 조회할 수 있다. 고객사 파일 원본 열람은 별도 권한과 감사 로그 기준을 따른다.

## 검증 방법

일반 repository cleanup에서는 DB/R2 mutation을 실행하지 않는다. 현재 기준 확인은 정적 참조, 계약 테스트, build, mutation audit, wrapper profile을 우선한다.

관련 검증은 다음 경로를 확인한다.

- `tools/pipeline/approved-workflow.ps1 -Action Verify -Profile repository-cleanup`
- `tests/functions-storage-contract.mjs`
- `tests/system-storage-usage-real-data-contract.mjs`
- `tests/simulator-adapter-plan-contract.mjs`
- `npm run test:smoke:db-api`는 승인된 개발/테스트 DB에서만 실행한다.

## 알려진 미해결 사항

- 실제 PG 결제 연동, 결제수단 저장, webhook 반영은 아직 별도 작업이다.
- R2 inventory와 DB metadata 간 reconciliation 표시는 추가 제품화가 필요하다.
- 작업지시서 첨부/디자인, PDF 산출물, 일반 문서함의 저장공간 포함/차단 기준은 route별로 확정해야 한다.
- 멤버 한도에서 비활성, 탈퇴 요청, 승인 대기 사용자를 포함할지는 최종 상품 정책 문구와 함께 확정해야 한다.

## 향후 작업

- 고객사 관리자 화면의 R2 reconciliation 안내 보강
- 시스템관리자 저장소 snapshot 운영 화면 검증
- 업로드 route별 quota guard 적용 범위 확정
- 요금제 변경 요청, 결제 실패, 해지 상태의 UX와 권한 정책 확정

## 폐기된 방식 주의

과거 버전별 billing/storage 문서는 구현 순서와 설계 이력을 담은 archive 자료다. 현재 판단은 이 문서와 실제 DB/API/repository/test 경로를 우선한다. 단순 R2 실시간 전체 스캔을 기본 사용량 기준으로 삼거나, 저장공간 초과를 이유로 기존 업무 데이터를 삭제하는 방식은 현재 기준이 아니다.
