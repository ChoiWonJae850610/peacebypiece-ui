# 발주 요청 이후 상태 설계

기준 버전: 0.16.49

## 목적

작업지시서 발주 요청 이후의 상태를 작업지시서 자체 상태, 원단·부자재 조달 상태, 발주 기록 상태로 분리한다. 화면에서 문자열을 직접 비교하지 않고, 타입과 정책 함수를 통해 상태 전이를 판단한다.

이 문서는 0.16.20 기준의 설계 문서이며, 실제 DB schema 변경이나 상태 변경 API 연결은 포함하지 않는다.

## 상태를 분리하는 이유

작업지시서 상태와 원단·부자재 발주 상태를 하나의 값으로 합치면 다음 문제가 생긴다.

- 원단 하나가 입고 지연되어도 작업지시서 전체 상태가 과도하게 흔들린다.
- 생산 진행 상태와 자재 조달 상태가 섞인다.
- 발주 요청 이력, 현재 발주 상태, 작업지시서 workflow가 서로 덮어쓰기 된다.
- 나중에 부분 발주, 부분 입고, 재발주, 대체 자재 처리가 어려워진다.

따라서 상태는 다음 세 계층으로 나눈다.

```text
workorder workflow status
material line procurement status
order request / purchase document status
```

## 상태 계층

### 1. 작업지시서 workflow status

작업지시서 자체의 업무 흐름을 나타낸다.

예상 범위:

- draft
- review_requested
- review_approved
- order_requested
- in_production
- inspection_requested
- completed
- archived

원칙:

- 작업지시서의 큰 업무 단계만 표현한다.
- 개별 원단·부자재 입고 상태를 직접 표현하지 않는다.
- 화면에서는 문자열 literal을 직접 비교하지 않는다.
- 상태 전이는 lib/workorder/policy 또는 service 계층의 함수에서만 판단한다.

### 2. material line procurement status

작업지시서에 연결된 원단·부자재 한 줄의 조달 상태를 나타낸다.

초기 후보:

- planned: 필요 항목으로 계획됨
- requested: 발주 요청 대상에 포함됨
- ordered: 실제 발주 완료
- partially_received: 일부 입고
- received: 입고 완료
- substituted: 대체 자재 사용
- canceled: 취소

원칙:

- workorder_material_lines 단위로 관리한다.
- 같은 작업지시서 안에서도 원단 A와 부자재 B의 상태는 다를 수 있다.
- 작업지시서 workflow status를 자동으로 덮어쓰지 않는다.
- 발주 가능 권한이 있는 사용자만 requested 이후 상태로 변경할 수 있다.

### 3. order request / purchase document status

발주 요청서, PDF, 외부 전달 문서, 공급처별 발주 기록의 상태를 나타낸다.

초기 후보:

- draft
- previewed
- requested
- sent
- confirmed
- failed
- canceled

원칙:

- 발주 문서 또는 공급처별 발주 묶음 단위로 관리한다.
- PDF/메일/카카오톡/문자 등 실제 전송 방식이 붙더라도 상태 의미는 유지한다.
- 발주 당시의 업체명, 품목명, 규격, 단가, 수량 등은 snapshot으로 보존할 수 있다.

## 권한 기준

권한은 다음처럼 나눈다.

```text
작업지시서 작성 가능:
- 작업지시서 생성/수정/삭제
- 작업지시서에 원단·부자재 필요 항목 연결/해제
- planned 상태의 기본 입력

발주 가능:
- 발주 요청 생성
- material line을 requested/ordered 계열로 변경
- 공급처별 발주 문서 상태 변경

조회 권한:
- 본인 담당 작업지시서 또는 허용된 범위의 상태 조회
```

화면 버튼 표시 조건과 API guard 조건은 같은 capability 함수를 기준으로 한다.

## 상태 전이 원칙

상태 전이는 다음 규칙을 따른다.

- 화면 컴포넌트에서 상태 문자열을 직접 비교하지 않는다.
- 상태 변경 가능 여부는 policy/capability 함수로 판단한다.
- service 계층에서 서버 기준으로 다시 검증한다.
- repository는 company_id, workorder_id scope를 강제한다.
- 상태 전이 실패는 공통 API 에러 형식으로 반환한다.

예상 함수:

```text
canRequestMaterialOrder(context)
canMarkMaterialOrdered(context)
canReceiveMaterialLine(context)
canCancelMaterialLineOrder(context)
getNextMaterialOrderStatuses(currentStatus, context)
```

## DB 방향

0.16.20에서는 DB를 변경하지 않는다. 이후 변경 후보는 다음과 같다.

```text
workorder_material_lines.order_status
workorder_material_lines.requested_at
workorder_material_lines.ordered_at
workorder_material_lines.received_at
workorder_material_lines.canceled_at
workorder_material_lines.status_updated_by

material_order_requests
material_order_request_lines
```

이미 0.16.18에서 workorder_material_lines는 기본 연결 테이블로 추가되었다. 발주 기록을 별도 테이블로 둘지, material line 상태만 먼저 확장할지는 0.16.21 이후 실제 화면 흐름을 기준으로 결정한다.

## snapshot 허용 기준

발주 이후 데이터는 현재 기준정보와 당시 발주 기록을 분리해야 한다.

허용되는 snapshot:

- 발주 당시 업체명
- 발주 당시 품목명
- 발주 당시 규격/색상/단위
- 발주 당시 수량/단가/금액
- 발주 당시 요청사항
- 발주서 PDF 생성 시점의 표시값

허용하지 않는 중복 필드:

- 현재 조회 화면에서만 쓰기 위한 company_name 복사값
- 기준정보에서 조인 가능한 최신 material name 복사값
- 인덱스로 해결 가능한 검색/정렬용 중복 상태값
- 원본과 불일치 가능성이 높은 편의용 count/cache 필드

## 통계/저장소/감사로그 연결 방향

- 원단·부자재 상태 변경은 나중에 감사로그 대상이 될 수 있다.
- 저장소 사용량에는 원단·부자재 텍스트 데이터 자체는 포함하지 않는다.
- 발주서 PDF 또는 첨부 파일이 생성될 경우에만 R2 저장소 사용량과 연결한다.
- 통계는 초기에는 실시간 조인 기준으로 계산하고, 병목이 확인되기 전에는 별도 summary table을 만들지 않는다.

## 0.16.21 이후 권장 순서

```text
1. material order status 타입/상수 확정
2. 상태 배지와 상태 변경 UI 추가
3. 상태 변경 API/service/repository 추가
4. 발주 가능 권한과 API guard 연결
5. 감사로그 후보 정리
6. 발주 문서/공급처별 요청 테이블 필요 여부 결정
```

## 비적용 범위

0.16.20에서 하지 않는 것:

- DB schema 변경
- full_reset.sql 변경
- 작업지시서 workflow 변경
- 발주 요청 API 변경
- PDF 생성 흐름 변경
- R2/첨부/메모/휴지통/purge 흐름 변경
