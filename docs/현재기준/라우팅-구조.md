# Routing Architecture

기준 버전: 0.16.49
상태: route baseline
목적: public, workspace, system, me 영역의 URL 책임과 보호 기준을 고정한다.

## 1. route 영역 구분

```txt
public    = 비로그인 접근 가능 영역
workspace = 고객사 업무공간
system    = 시스템관리자 운영공간
me        = 로그인 사용자 개인설정
api       = 서버 요청 진입점
```

## 2. 목표 URL 구조

고객사 업무공간은 `/workspace`를 기준으로 한다.

```txt
/workspace
/workspace/workorders
/workspace/workorders/[id]
/workspace/partners
/workspace/files
/workspace/stats
/workspace/members
/workspace/settings
/workspace/materials
```

시스템관리자 영역은 `/system`을 기준으로 유지한다.

```txt
/system
/system/customers
/system/storage-usage
/system/audit-logs
/system/billing
/system/policies
```

개인설정은 `/me`를 기준으로 유지한다.

```txt
/me/settings
```

## 3. /admin 경로 정리 기준

`/admin`은 고객사 업무공간 의미와 충돌하므로 `/workspace`로 전환한다.

```txt
- 신규 내부 링크는 /workspace를 사용한다.
- /system 경로는 /workspace로 이동하지 않는다.
- /me/settings는 workspace 설정과 분리한다.
- legacy /admin redirect는 기본적으로 두지 않는다.
- 남은 /admin 화면 링크는 단계적으로 제거한다. API 경로와 admin 접두어 공통 컴포넌트는 0.16.24 기준 별도 rename 후보로 분리한다.
```

## 4. route guard 기준

route guard는 주소 접근 가능 여부만 판단한다.

```txt
/workspace/* = 로그인 + 고객사 소속/승인 상태 필요
/system/*    = 로그인 + 시스템관리자 필요
/me/*        = 로그인 필요
/public      = 별도 보호 없음
```

route guard가 담당하지 않는 것:

```txt
- 버튼 표시 조건
- 상세 액션 가능 여부
- API 직접 호출 권한
- DB companyId scope
```

각 책임은 다음 계층에서 분리한다.

```txt
route guard    = 이 주소에 들어갈 수 있는가
capability     = 이 화면에서 이 버튼을 누를 수 있는가
API guard      = 이 요청을 서버에서 실행해도 되는가
repository     = 이 회사 데이터만 조회/변경하는가
```

## 5. 직접 주소 접근 기준

직접 URL 입력 시에도 보호 기준이 동일해야 한다.

```txt
- 비로그인 사용자가 /workspace 접근 시 보호 처리
- 비로그인 사용자가 /system 접근 시 보호 처리
- 비로그인 사용자가 /me/settings 접근 시 보호 처리
- 일반 멤버가 /system 접근 시 차단
- 권한 없는 접근에서 빈 화면/무한 로딩 금지
```

## 6. workspace shell 기준

workspace shell은 고객사 업무공간 전용이다.

기본 메뉴 후보:

```txt
- 작업지시서
- 협력업체관리
- 저장소관리
- 통계정보
- 멤버관리
- 환경설정
- 원단·부자재
```

규칙:

```txt
- 메뉴 href는 /workspace 기준으로 통일한다.
- 시스템관리자 메뉴와 섞지 않는다.
- 개인설정은 사람 아이콘으로 분리한다.
- 톱니바퀴는 관리자 설정 의미로 남긴다.
```

## 7. system shell 기준

system shell은 서비스 운영자 전용이다.

```txt
- 고객사 관리
- 저장소 사용량/삭제 처리 로그
- 감사 로그
- 요금/청구/결제 정책
- 약관/개인정보/운영 정책
```

규칙:

```txt
- system 화면은 고객사 업무 메뉴와 섞지 않는다.
- system API는 시스템관리자 guard를 별도로 둔다.
- 고객사 데이터 접근이 필요한 경우 접근 사유/감사 로그 정책을 따른다.
```

## 8. API route 기준

API route는 URL 진입점이며 업무 로직을 길게 두지 않는다.

```txt
app/api/*/route.ts
→ API guard
→ service 호출
→ response 변환
```

금지:

```txt
- route.ts 안에 DB query 누적
- route.ts 안에 상태 전이 로직 작성
- route.ts 안에 R2 key 조립 로직 작성
- 화면 권한과 다른 서버 권한 기준 사용
```

## 9. i18n / copy 기준

라우팅 변경으로 사용자에게 보이는 명칭이 바뀌는 경우 i18n key를 확인한다.

```txt
- 고객사 업무공간 = workspace 의미
- 시스템관리자 = system 의미
- 개인설정 = me 의미
- 기술 경로명이 사용자 문구에 노출되지 않게 한다
```

## 10. 0.16.25 기준 라우팅 안정화 상태

0.16.25 기준 현재 고객사 업무 화면은 `/workspace` route group 하위에서 운영한다.

```txt
현재 화면 route:
- /workspace
- /workspace/workorders
- /workspace/materials
- /workspace/partners
- /workspace/files
- /workspace/stats
- /workspace/members
- /workspace/settings
- /workspace/standards
- /workspace/invites
- /workspace/subscription
- /workspace/units
- /workspace/history
```

현재 시스템관리자 화면은 `/system` route group 하위에서 운영한다.

```txt
현재 system route:
- /system
- /system/companies
- /system/storage-usage
- /system/audit-logs
- /system/billing
- /system/policies
- /system/invites
- /system/standards
- /system/category-rules
- /system/access-checkpoint
```

주의할 미정리 영역은 API 경로다. `/api/admin/*`는 0.16.25 기준 아직 삭제하거나 rename하지 않는다. 화면 URL과 API URL rename을 같은 단계에서 처리하면 저장소/휴지통/멤버/설정 회귀 위험이 크므로 후속 버전에서 alias → client 이동 → 제거 순서로 처리한다.
