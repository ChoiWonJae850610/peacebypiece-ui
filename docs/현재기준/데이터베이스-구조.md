# Database Architecture Policy

기준 버전: 0.16.49
기준 원본: 0.16.47
범위: DB 구조, payload/json, 중복 컬럼, snapshot, 조인/인덱스, full_reset 정리 기준

## 1. 목적

이 문서는 WAFLOW / PeaceByPiece의 DB 구조를 기능 개발 전에 다시 흐트러지지 않게 고정하기 위한 기준 문서다.

0.16.15의 핵심은 단순히 payload/json을 줄이는 것이 아니다. 다음 항목을 함께 본다.

- payload/json에 숨은 업무 데이터
- 조인을 줄이려고 복사 저장한 표시용 필드
- 인덱스로 해결 가능한데 중복 저장된 필드
- 현재 화면, API, repository에서 쓰지 않는 legacy 컬럼
- 당시 값 보존이 필요한 snapshot 필드
- 현재 상태 조회용 테이블과 이력/snapshot 테이블의 경계

## 2. 기본 원칙

```txt
현재 상태 조회용 데이터:
- 원본 테이블 + FK + 인덱스 + 명확한 repository query로 처리한다.
- 조인을 피하려고 표시명/상태/회사명/거래처명을 무조건 복사 저장하지 않는다.

당시 기록 보존용 데이터:
- 발주서, PDF, 감사로그, 이력, 과거 상태 증빙처럼 당시 값 보존이 필요한 경우 snapshot을 허용한다.
- snapshot 컬럼은 현재 조회용 중복 컬럼과 이름/목적을 분리한다.

검색/통계/권한/상태 판단 데이터:
- payload/json 안에 두지 않는다.
- 컬럼, FK, enum/check constraint, summary table 중 하나로 구조화한다.

화면 편의용 중복 표시값:
- 기본적으로 금지한다.
- 필요하면 snapshot 목적, 성능 병목, 외부 문서 증빙 중 하나를 명확히 문서화한다.

성능 최적화:
- 실제 병목 확인 전 denormalization을 먼저 하지 않는다.
- 먼저 company_id, spec_sheet_id, partner_id, member_id, status, created_at 기준 인덱스와 query plan을 점검한다.
```

## 3. payload/json 사용 기준

### 허용

```txt
- audit_logs.metadata: 감사 이벤트의 보조 metadata
- history_logs.metadata: 사용자에게 보여줄 이력의 보조 metadata
- 외부 요청의 typed detail 중 schema가 아직 안정되지 않은 제한적 보조값
- 통계 snapshot의 재계산 보조 metadata
```

단, 위 항목도 아래 값은 저장하지 않는다.

```txt
- raw request body 전체
- raw token
- password, secret, API key
- cookie, authorization header
- 실제 R2 signed URL
- 전체 카드번호, CVC, 카드 비밀번호, 주민등록번호, 카드 유효기간 원문
- UI 표시 문장만 있는 status/reason 값
```

### 금지

```txt
- 권한 판단 값
- workflow status
- 작업지시서 상태
- 발주 상태
- 검수 상태
- 삭제/복원/purge 상태
- 통계 group key
- 검색 필터 key
- company/member/workorder scope key
```

이 값들은 query, permission, 통계, 감사, 접근 차단에 영향을 주므로 컬럼화해야 한다.

## 4. 중복 컬럼 판단 기준

중복 컬럼은 아래 순서로 판단한다.

```txt
1. 원본 FK가 있는가?
2. 현재 화면에서 매번 최신 원본명을 보여줘야 하는가?
3. 과거 문서/이력에서 당시 이름을 보존해야 하는가?
4. 회사명/거래처명 변경 시 과거 데이터가 바뀌면 안 되는가?
5. 현재 조회 성능 병목이 실제로 확인되었는가?
6. 같은 값이 여러 테이블에서 업데이트 불일치를 만들 가능성이 있는가?
```

판정:

```txt
- 최신 원본명이 필요하면 FK + join.
- 당시 값 보존이 필요하면 snapshot 컬럼.
- 성능만 이유라면 먼저 인덱스/쿼리 점검.
- 화면 label만 이유라면 presentation/i18n에서 생성.
```

## 5. snapshot 허용 기준

snapshot은 허용하되, 이름과 목적을 명확히 해야 한다.

허용 예:

```txt
- 발주서/PDF 생성 당시의 업체명, 요청사항, 수량, 금액
- 감사로그 이벤트 당시의 actor role, target summary
- 삭제/복원/purge 당시의 대상 수량과 batch id
- 요금제 변경 당시의 plan code, price, storage limit
```

주의 예:

```txt
- spec_sheets.company_name
- attachments.company_name
- memos.company_name
- material_stocks.company_name
- orders.factory_name
- spec_sheet_materials.vendor
- spec_sheet_outsourcing_lines.vendor
```

위 필드는 무조건 삭제 대상이라는 뜻이 아니다. 다만 “현재 조회용 복사값”인지 “당시 표시 보존 snapshot”인지 명확하지 않으면 정리 후보로 둔다.

## 6. 현재값 테이블과 이력 테이블 분리

현재값 테이블:

```txt
- spec_sheets
- orders
- spec_sheet_materials
- spec_sheet_outsourcing_lines
- material_stocks
- company_members
- member_permissions
```

이 테이블은 현재 상태 조회와 업무 진행을 위해 사용한다.

이력/snapshot 테이블:

```txt
- audit_logs
- history_logs
- storage_usage_snapshots
- company_workorder_daily_stats
- company_workorder_monthly_stats
- company_storage_daily_stats
- 향후 order_request_snapshots 또는 generated_document_snapshots 후보
```

과거 시점 보존이 필요하면 현재값 테이블에 임의 컬럼을 계속 붙이지 말고 snapshot/history 계층으로 분리한다.

## 7. full_reset 기준

개발 단계에서는 기존 데이터 보존보다 구조 명확성이 우선이다. 따라서 full_reset을 수정할 때는 아래를 따른다.

```txt
- legacy compatibility 컬럼은 기본적으로 유지하지 않는다.
- 삭제 후보 컬럼은 repository/API/UI 사용 여부를 먼저 확인한다.
- 삭제 전 docs/보관문서/점검기록/데이터베이스-컬럼-점검.md에 후보와 근거를 기록한다.
- index 추가/삭제는 docs/현재기준/데이터베이스-쿼리-인덱스-정책.md 기준을 따른다.
- smoke test는 schema 존재 여부뿐 아니라 legacy 컬럼 잔존 여부도 확인한다.
```

## 8. 당장 삭제하지 않는 이유

0.16.15는 schema 변경 버전이 아니다. 이 버전에서는 기준을 문서화한다.

이유:

```txt
- 현재 작업지시서/R2/첨부/메모/휴지통/purge 흐름이 동작 중이다.
- 중복 컬럼 중 일부는 snapshot으로 유지할 수 있다.
- 삭제 전 repository mapper와 UI 표시 흐름을 확인해야 한다.
- full_reset.sql, seed, smoke test, repository 변경은 별도 버전으로 분리하는 편이 안전하다.
```

다음 schema 변경 버전에서는 이 문서의 기준에 따라 컬럼 제거/이름 변경/인덱스 조정을 작은 단위로 진행한다.

## 13. 0.16.25 안정화 기준

0.16.25 기준 DB 구조 정리 원칙은 다음으로 고정한다.

```txt
- 기준정보성 원단·부자재 데이터는 materials 계열 테이블에 둔다.
- 작업지시서와 원단·부자재의 연결은 workorder_material_lines 계층에 둔다.
- 발주 상태는 작업지시서 workflow status와 분리해서 material line order_status로 관리한다.
- 실제 발주서/PDF에 넣을 당시 값 보존은 후속 snapshot 정책에서 별도 결정한다.
- 현재 조회용 표시명 중복 저장은 기본적으로 금지하고 FK + index + repository query를 우선한다.
- 통계 병목이 확인되기 전 summary/cache table을 먼저 만들지 않는다.
```

후속 DB 변경이 필요한 경우에는 `full_reset.sql`, `full_reset_smoke_test.sql`, service/repository, 문서를 같은 버전에서 맞춘다.
