# 원단·부자재 발주 기능 설계

- 기준 버전: `0.16.90`
- 문서 성격: 원단·부자재 발주 업무의 1차 설계 기준
- 적용 범위: 업무 정의, 화면 IA, 상태 흐름, 권한 흐름, DB 초안, 작업지시서 연결, 자재 전달 요청서/PDF 확장 방향
- 비적용 범위: 실제 DB schema 반영, API 구현, 화면 구현, PDF 생성 구현

## 1. 기능 정의

원단·부자재 화면은 단순한 기준정보 관리 화면이 아니라 고객사가 작업지시서 생산에 필요한 자재를 요청, 검토, 발주, 배분, 재고화하는 업무 화면이다.

핵심 목적은 다음 네 가지를 분리해서 기록하는 것이다.

1. 작업지시서 필요량
2. 실제 발주 수량
3. 작업지시서별 배분 수량
4. 남은 재고 수량

필요량과 발주량은 항상 같지 않다. 발주 담당자는 공급 단위, 여유분, 예비분, 최소 발주 수량 때문에 필요량보다 많이 주문할 수 있다. 이때 작업지시서에 배분되지 않은 수량은 고객사 재고로 남아야 한다.

예시:

- A 작업지시서: 30수 면 1마 필요
- B 작업지시서: 30수 면 1마 필요
- 실제 발주: 10마
- 배분:
  - A 작업지시서 1마
  - B 작업지시서 2마
  - 재고 7마

위 예시에서 B 작업지시서의 배분 수량은 필요량보다 클 수 있다. 이 초과분은 예비분 또는 작업 손실 대비분으로 해석한다.

## 2. 업무홈 IA 위치

업무 대시보드의 권장 순서는 다음과 같다.

1. 작업지시서
2. 원단·부자재
3. 협력업체관리
4. 통계정보
5. 기준정보
6. 저장소관리
7. 환경설정
8. 멤버관리

원단·부자재는 작업지시서 바로 다음에 배치한다. 생산 흐름상 작업지시서 작성 이후 바로 자재 발주 및 배분이 이어지기 때문이다.

## 3. MVP 화면 구성

### 3.1 상단 요약 카드

- 작성중 발주 요청
- 검토 대기
- 발주 완료
- 현재 재고 품목 수
- 이번 달 자재 구매액

### 3.2 탭 구성

1. 발주 요청 목록
2. 발주 작성/상세
3. 재고 현황

### 3.3 발주 작성/상세 화면

필수 입력 또는 표시 항목:

- 공급처 선택
- 품목 추가
- 작업지시서 연결
- 품목별 주문 수량 입력
- 작업지시서별 배분 수량 입력
- 주문 수량 - 배분 수량 = 재고 예정 수량 표시
- 검토 요청 버튼
- 바로 발주 버튼

## 4. 상태 흐름

1차 상태 후보:

- `draft`: 작성중
- `review_requested`: 검토요청
- `approved`: 승인/발주확정
- `ordered`: 발주완료
- `rejected`: 반려
- `canceled`: 취소

확장 상태 후보:

- `delivery_requested`: 전달 요청 완료
- `received`: 입고
- `inventory_posted`: 재고 반영

초기 구현에서는 발주 업무와 재고 반영 업무를 과하게 합치지 않는다. 발주 확정 후 재고 lot 생성은 별도 flow로 두고, 입고 검수나 실제 재고 반영은 이후 확장한다.

## 5. 권한 흐름

권한 코드 후보:

- `material.read`: 원단·부자재 조회
- `material.order.request`: 자재 발주 요청 작성 및 검토 요청
- `material.order.place`: 검토 없이 바로 발주 또는 검토 요청 승인
- `material.inventory.read`: 자재 재고 현황 조회

역할별 1차 정책:

| 역할 | 허용 범위 |
| --- | --- |
| 디자이너 | 자재 발주 요청서 작성, 작업지시서 연결, 검토 요청 |
| 발주 권한자 | 검토 없이 바로 발주, 검토 요청 승인 |
| 고객사 관리자 | 전체 자재 발주 조회, 승인/반려/발주 처리, 재고 현황 조회 |
| 조회 전용 | 조회만 가능 |

권한 판단은 TSX 내부 문자열 비교로 직접 처리하지 않는다. 기존 권한 리팩토링 방향에 맞춰 permission code, policy helper, viewModel로 분리한다.

## 7. 작업지시서 연결 방식

자재 발주 line은 여러 작업지시서에 배분될 수 있다. 반대로 하나의 작업지시서는 여러 자재 발주 line과 연결될 수 있다.

관계 기준:

- `material_order_lines` 1건은 `material_order_allocations` 여러 건을 가진다.
- `material_order_allocations.work_order_id`는 작업지시서 id를 참조한다.
- 배분 수량 합계는 주문 수량을 초과할 수 없도록 서버 검증이 필요하다.
- 배분 수량이 작업지시서 필요량을 초과하는 것은 허용한다.
- 주문 수량에서 배분 수량 합계를 뺀 값은 재고 예정 수량으로 계산한다.

## 8. 자재 전달 요청서 / PDF 방향

발주 확정 이후에는 작업지시서 PDF와 별개의 자재 전달 요청서를 생성한다.

문서는 두 종류로 분리한다.

1. 내부용 발주서
   - 단가 포함
   - 총액 포함
   - 공급처별 발주 관리 목적
2. 외부 전달용 자재 전달 요청서
   - 단가 제외
   - 총액 제외
   - 픽업지, 도착지, 품목, 수량, 작업지시서 구분 중심

PDF 생성 기준은 작업지시서 개수 기준보다 `공급처 + 도착지` 기준이 실무적으로 적합하다.

예시:

- A 원단 → 봉제공장 1
  - 작업지시서 A 2마
  - 작업지시서 B 3마
- A 원단 → 봉제공장 2
  - 작업지시서 C 1마
- A 원단 → 고객사 보관
  - 재고 4마

## 12. 구현 시 주의사항

- 화면 TSX에서 상태 문자열, 역할 문자열, 권한 문자열을 직접 비교하지 않는다.
- 발주 상태, 권한 코드, action code는 상수 또는 policy helper로 분리한다.
- API route는 thin하게 유지하고 실제 검증/계산은 lib 계층으로 분리한다.
- DB schema 변경 시 `full_reset.sql`과 smoke test SQL 영향을 같이 확인한다.
- 기존 작업지시서/R2/첨부/메모/휴지통/purge 흐름은 직접 목표가 아니면 건드리지 않는다.
- 단가/총액이 포함되는 내부용 문서와 단가/총액을 제외하는 외부 전달 문서를 반드시 분리한다.
