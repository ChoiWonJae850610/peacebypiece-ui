# 39. 서비스 운영 IA

## 1. 목적

0.15.18은 WAFL을 실제 SaaS로 운영하기 위한 시스템관리자와 고객사관리자 정보 구조를 정리한다.

이 문서는 화면 구현 전에 운영 메뉴, 책임 범위, 고객사 과금·증빙·서비스 문서 노출 기준을 고정하기 위한 설계 문서다.

```txt
적용 성격:
- IA/정책 문서화
- route 후보 정리
- 권한/상태/데이터 책임 범위 정리

이번 버전 제외:
- 기능 코드 구현
- DB schema 생성
- API 구현
- R2 처리 변경
- 실제 PG 연동
```

## 2. 시스템관리자 운영 IA

시스템관리자는 개별 고객사의 업무 데이터를 직접 운영하는 화면이 아니라, 고객사 계정·구독·결제·증빙·정책·감사로그를 관리하는 운영 콘솔이다.

```txt
최종 메뉴 후보:
1. 고객사 관리
2. 요금제 관리
3. 고객사 구독 관리
4. 결제수단 관리
5. 결제내역
6. 결제 실패/미납 관리
7. 환불 관리
8. 증빙 관리
9. 정산자료 출력
10. 서비스 문서 관리
11. 시스템 감사로그
```

### 2.1 고객사 관리

```txt
책임:
- 고객사 목록 조회
- 고객사 관리자 초대
- 가입 신청 승인/거절/재입력 요청
- 고객사 상태 변경
- 서비스 중지/재개

초기 route 후보:
- /system/companies
- /system/companies/[companyId]
- /system/companies/[companyId]/approval
```

### 2.2 요금제 관리

```txt
책임:
- 요금제 생성/수정
- 저장공간 한도 관리
- 월 요금 관리
- 사용/미사용 상태 관리
- trial 정책 관리

초기 route 후보:
- /system/billing/plans
```

요금제 관리는 고객사별 구독 상태와 분리한다. 요금제는 상품 정의이고, 고객사 구독은 특정 고객사에 적용된 계약 상태다.

### 2.3 고객사 구독 관리

```txt
책임:
- 고객사별 현재 요금제 조회
- trial 기간 조회/조정
- 구독 상태 조회/변경
- 다음 결제일 조회
- 서비스 중지/재개

초기 route 후보:
- /system/billing/subscriptions
- /system/companies/[companyId]/subscription
```

### 2.4 결제수단 관리

```txt
책임:
- PG사 확인
- customerKey 확인
- billingKey 상태 확인
- 카드사/카드 마지막 4자리 표시
- 결제수단 상태 변경 이력 조회

초기 route 후보:
- /system/billing/payment-methods
```

시스템은 전체 카드번호, CVC, 카드 비밀번호, 주민등록번호, 카드 유효기간 원문을 저장하지 않는다.

### 2.5 결제내역

```txt
책임:
- PG 거래번호 조회
- 승인번호 조회
- 결제일 조회
- 결제금액 조회
- 결제상태 조회
- 영수증 URL 조회

초기 route 후보:
- /system/billing/payments
```

### 2.6 결제 실패/미납 관리

```txt
책임:
- 결제 실패 고객사 목록
- 실패 사유 확인
- 재결제 필요 여부 확인
- 서비스 제한 여부 확인
- 미납 상태 확인

초기 route 후보:
- /system/billing/failures
- /system/billing/overdue
```

### 2.7 환불 관리

```txt
책임:
- 환불 요청 조회
- 환불 사유 확인
- 환불 금액 확인
- PG 환불 거래번호 확인
- 환불 상태 관리

초기 route 후보:
- /system/billing/refunds
```

### 2.8 증빙 관리

```txt
책임:
- 카드매출전표 조회
- 세금계산서 요청 상태 관리
- 현금영수증 후보 보관
- 수동증빙 등록 여부 관리
- 발행 상태 관리

초기 route 후보:
- /system/billing/evidence
```

초기 제품은 카드결제 중심이므로 기본 증빙은 카드매출전표다. 사업자 고객사의 세금계산서 요청 여부는 별도 필드로 관리한다.

### 2.9 정산자료 출력

```txt
책임:
- 기간별 매출 조회
- 공급가액/부가세/합계 조회
- 고객사별 매출 조회
- 증빙 발행 상태 조회
- 출력/다운로드 후보 제공

초기 route 후보:
- /system/billing/settlements
```

### 2.10 서비스 문서 관리

```txt
책임:
- 이용약관 관리
- 개인정보처리방침 관리
- 환불정책 관리
- 데이터 보관/삭제정책 관리
- 요금제 정책 관리
- 공지사항 관리

초기 route 후보:
- /system/legal
- /system/notices
```

서비스 문서는 시스템관리자가 수정하고, 고객사관리자는 읽기 전용으로 확인한다.

### 2.11 시스템 감사로그

```txt
책임:
- 고객사 승인/거절 기록
- 요금제 변경 기록
- 결제/환불 처리 기록
- 저장소 purge 기록
- 권한 변경 기록
- 시스템 설정 변경 기록

초기 route 후보:
- /system/audit-logs
```

감사로그는 고객사 업무 히스토리와 분리한다. 고객사가 보는 최소 이력과 시스템 운영자가 보는 상세 감사로그는 같은 화면으로 합치지 않는다.

## 3. 고객사관리자 운영 IA

고객사관리자는 본인 회사의 업무 데이터, 결제/증빙 정보, 서비스 문서를 확인한다. 시스템관리자 수준의 고객사 상태 변경, 요금제 상품 정의, 환불 처리 권한은 갖지 않는다.

```txt
최종 메뉴 후보:
1. 현재 요금제
2. 카드 등록/변경
3. 결제내역
4. 영수증 보기
5. 사업자 정보
6. 증빙 요청 정보
7. 이용약관 보기
8. 개인정보처리방침 보기
9. 환불정책 보기
10. 데이터 보관/삭제정책 보기
```

### 3.1 추천 route 구조

```txt
/admin/settings
- 회사 정보
- 관리자 계정
- 기준정보 설정
- 개발 건의

/admin/billing
- 현재 요금제
- 카드 등록/변경
- 결제내역
- 영수증
- 증빙 요청 정보

/admin/legal
- 이용약관
- 개인정보처리방침
- 환불정책
- 데이터 보관/삭제정책
```

### 3.2 고객사관리자 홈 카드 후보

```txt
- 작업지시서 업무 화면
- 원단/부자재 발주
- 협력업체 관리
- 저장소 관리
- 통계정보
- 멤버 관리
- 결제/증빙
- 서비스 문서
```

결제/증빙과 서비스 문서는 업무 핵심 카드와 같은 비중으로 노출하지 않아도 된다. 운영 초기에는 환경설정 또는 별도 billing/legal 진입으로 분리한다.

## 4. 메뉴 책임 분리 원칙

```txt
시스템관리자:
- 고객사 계정/상태 관리
- 요금제 상품 관리
- 구독/결제/환불/증빙 운영
- 서비스 문서 발행
- 시스템 감사로그 확인

고객사관리자:
- 본인 회사 업무 데이터 관리
- 본인 회사 결제수단 등록/변경
- 본인 회사 결제내역/영수증 확인
- 본인 회사 사업자 정보 관리
- 서비스 문서 읽기

일반 멤버/디자이너:
- 권한에 맞는 작업지시서 업무
- 권한에 맞는 원단/부자재 발주 업무
- 개인설정
```

## 5. 0.15.19 구현 연결 기준

0.15.19에서 화면 메뉴를 연결할 경우 다음 순서가 안전하다.

```txt
1. /system 홈 운영 카드 재배치
2. 미구현 route는 준비 중 상태로 표시
3. /admin 홈 또는 /admin/settings에 billing/legal 진입 카드 추가
4. 실제 PG/DB/API 없이 static IA만 반영
5. i18n key 후보를 먼저 정리
```
