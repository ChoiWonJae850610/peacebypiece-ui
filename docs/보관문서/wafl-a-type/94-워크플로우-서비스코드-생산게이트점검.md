# 94. 작업지시서 workflow / serviceCode / production sync gate 점검

## 기준 버전

- 기준: 0.15.81
- 결과: 0.15.82

## 점검 배경

0.15.73.3~0.15.81 구간에서 작업지시서 생성, 상세 hydration, draft 보존, 발주정보 저장 정책을 순차적으로 보정했다. 이후 workflow action 실행 시점에 다음 세 흐름이 같은 기준을 보도록 정리할 필요가 있었다.

- workflow validation
- workflow serviceCode
- production composition 저장 gate

## 정리 방향

### 1. workflow validation gate

`lib/workorder/workflowActionGate.ts`를 추가하여 workflow action 실행 직전의 검증 경로를 모았다.

- 검토요청 검증
- 검토완료 검증
- 발주요청 검증
- workflow 검증 전 생산구성 numeric 값 정규화
- workflow serviceCode 계산

기존에는 `useWorkOrderWorkflowActions.ts` 안에서 검토요청/검토완료/발주요청 검증이 직접 흩어져 있었다. 이번 정리는 hook에서 직접 판단하던 검증 일부를 domain utility로 이동하는 1차 정리다.

### 2. production composition snapshot 정규화

workflow 검증 전 `normalizeProductionCompositionForWorkflowSnapshot()`을 사용한다.

목표:

- 발주정보 수량/공임/손실비 숫자값 정규화
- 원단/부자재 수량/단가/합계 정규화
- 외주 수량/단가/합계 정규화
- 검토요청/발주요청 직전 검증이 화면 draft와 다른 값을 보지 않도록 방지

### 3. serviceCode 연결

workflow action serviceCode는 `getServiceCodeForWorkflowAction()`으로 모아 계산한다.

- 검토요청: `WO-F001`
- 검토완료: `WO-F002`
- 발주요청: `WO-F003`
- 검수완료: `WO-F004`
- 완료처리: `WO-F005`
- 반려/취소/되돌리기 계열: backward/revert code

### 4. production composition 저장 gate

현재 production composition 저장 gate는 다음 경로를 유지한다.

```text
workflow action
→ getServiceCodeForWorkflowAction()
→ persistWorkOrderStatePatchWithHistory()
→ buildWorkOrderStatePatch()
→ shouldCommitProductionComposition()
→ guardProductionCompositionPatchByServiceCode()
```

즉, production composition은 serviceCode가 허용하는 경우에만 state patch에 포함된다.
