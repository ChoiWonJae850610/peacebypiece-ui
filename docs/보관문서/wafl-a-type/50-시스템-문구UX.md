# 50. WAFL A-TYPE system 화면 개발자성 문구 정리

Version: 0.16.49

## 목적

0.15.25~0.15.26에서 public/auth 화면의 개발자성 문구를 정리한 뒤, 시스템관리자 화면에 남아 있는 내부 구현 설명과 기획 문서형 문구를 운영자용 문장으로 완화한다.

## 적용 대상

```txt
- /system
- /system/companies
- /system/billing
- /system/invites
- /system/standards/seed-status
- /system/access-checkpoint
- 시스템관리자 i18n 문구
```

## 정리 기준

### 제거 또는 완화할 표현

```txt
- 개발용
- preview
- skeleton
- seed
- fallback
- DB/API/Worker/R2 등 구현 중심 설명
- company_users / permission_code / company_plan_assignments 같은 내부 테이블·코드명
- raw token / token_hash 같은 보안 구현 설명
- 0.10.x 기준, SQL 실행 안내, 내부 패치 파일명
- “설계 화면”, “연결될 영역”, “후속 연결”처럼 제품 화면보다 기획 문서에 가까운 문장
```

### 유지 가능한 표현

```txt
- 시스템관리자에게 실제로 필요한 운영 상태
- 고객사 승인/거절/재입력 요청
- 저장소 삭제 후보
- 요금제·용량 운영 기준
- 초대 링크 생성/복사/전달
- 기준정보 초기값 상태
```

## 화면별 반영 내용

### 시스템관리자 홈

```txt
변경:
- “설계 중”, “준비 중”, “관리할 영역” 표현을 “운영 예정”, “관리합니다” 중심으로 완화
- PG/customerKey/billingKey 같은 결제 내부 저장 표현을 운영자용 결제수단 문구로 변경
- purge 표현을 저장소 삭제 처리 문구로 완화

유지:
- 고객사 운영 / 요금·결제·증빙 / 시스템 운영 기준 IA
- 기존 route 연결
```

### 고객사 관리

```txt
변경:
- DB 파라미터 오류 문구를 사용자용 처리 실패 안내로 완화
- 이메일/SMS 후속 연결 문구를 링크 복사 안내로 정리
- 내부 구현 설명보다 지금 가능한 작업 중심으로 표현

유지:
- 초대 생성
- 승인/거절/재입력 요청
- 사업자등록증/통장사본 보기
- 승인 API 흐름
```

### 요금제·용량 관리

```txt
변경:
- preview, override, lib/billing, API 연결 전 등의 표현 제거 또는 완화
- 샘플 고객사/샘플 브랜드 표현을 운영 예시 고객사 표현으로 완화
- company_plan_assignments 등 내부 테이블명 노출 제거
- 비활성 버튼은 “저장 준비중”보다 “요금제 변경 저장”처럼 최종 액션명으로 표시

유지:
- 요금제 목록
- 고객사별 사용량/멤버 수
- 예외 한도 개념
- 저장 기능 비활성 상태
```

### 고객사 관리자 초대

```txt
변경:
- raw token / token_hash / DB 컬럼 / permission_code 설명 제거
- 초대 링크 생성, 복사, QR 전달 중심으로 사용자용 문구 정리
- 미리보기 표현을 “가입 신청 화면 확인”으로 완화

유지:
- 초대 링크/QR 정책
- 승인 전 고객사 미생성 정책
- 시스템관리자 승인 후 고객사 생성 흐름
```

### 기준정보 초기값 상태

```txt
변경:
- seed, DB mode, SQL 패치 실행 안내 제거
- 기준정보 기본값이 부족할 때 운영자가 이해할 수 있는 안내로 변경

유지:
- 단위/공정/생산품 유형 항목 수 확인
- 최소 기준 미달 상태 표시
```

### 접근 안정성 점검

```txt
변경:
- checkpoint 문구를 운영 점검 표현으로 완화
- owner 표기를 담당 영역으로 변경
- 실제 연결 후보 표현을 운영 보완 후보로 변경

유지:
- 화면별 안정 상태 요약
- route 이동 링크
```

## 변경하지 않은 것

```txt
- DB schema
- API route
- R2/Worker 처리 로직
- OAuth/세션/권한 판정
- 승인/거절/재입력 요청 동작
- 요금제 저장 기능 활성화 여부
- route 구조
```
