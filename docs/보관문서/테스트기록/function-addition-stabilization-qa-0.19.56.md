# 0.19.56 기능 추가 전 안정화 QA 기준 정리

## 목적

0.19.57 이후 기능 수정/추가를 시작하기 전에 작업지시서, 원단·부자재 발주, 저장소/R2, 첨부·디자인·메모, 권한별 버튼과 서버 거부 흐름을 한 번에 확인하기 위한 QA 기준을 정리한다.

이번 버전은 QA 문서와 버전 증가만 포함한다. 실제 기능, API, DB, R2, PDF, workflow 코드는 변경하지 않는다.

## 현재 판단

- 고객사 관리자 주요 UI 공통화는 1차 마감 상태로 볼 수 있다.
- 기능 추가 전에는 핵심 업무 흐름 회귀 테스트가 필요하다.
- 특히 작업지시서와 원단·부자재 발주는 상태, PDF, 첨부, 자재 할당이 서로 연결되어 있어 기능 추가 전에 기준선을 확정해야 한다.

## 공통 확인 기준

### 기본 환경

- PC Chrome
- iPhone 세로/가로
- iPad mini 세로/가로
- iPad Pro 세로/가로
- Galaxy Tab 세로/가로

### 공통 UI

- 화면 진입 시 폭 넘침이 없어야 한다.
- PageHero, SummaryCard, SectionPanel, DataTable, Button, ActionButton, Toast, Modal, Empty/Loading/Error 상태가 WAFL 기준으로 표시되어야 한다.
- Galaxy Tab 세로↔가로 반복 회전 후 stale width 문제가 재발하지 않아야 한다.
- 버튼 클릭 후 로딩/처리중/완료/실패 피드백이 보여야 한다.

## 작업지시서 QA

### 목록/상세

- 작업지시서 목록 조회가 회사 범위 기준으로 동작해야 한다.
- 일반 멤버는 본인 담당 작업지시서만 보여야 한다.
- 고객사 관리자는 회사 전체 작업지시서를 볼 수 있어야 한다.
- 목록 카드 선택 시 중앙 상세와 우측 첨부/디자인/메모 패널이 같은 작업지시서를 가리켜야 한다.
- 목록 카드 우측 action menu가 열리고 닫혀야 한다.

### 생성/수정/삭제

- 권한이 있는 사용자는 작업지시서를 생성할 수 있어야 한다.
- 권한이 없는 사용자는 UI 버튼이 제한되거나 서버에서 거부되어야 한다.
- 제목/담당자 변경은 즉시 반영되어야 한다.
- 삭제 시 휴지통 흐름으로 이동해야 한다.
- 삭제/복원/영구삭제 이후 목록과 상세 상태가 일관되어야 한다.

### 상태 변경

- 작성중 → 검토요청
- 검토요청 → 검토완료
- 검토완료 → 발주요청
- 발주요청 → 자재 발주 대기
- 자재 발주 완료 누적 → 납기 입력
- 검수 단계 → 완료

확인 기준:

- 권한이 없는 단계 버튼은 숨김/비활성/서버 거부 중 하나로 처리되어야 한다.
- 상태 변경 성공 후 목록/상세/진행단계 UI가 즉시 갱신되어야 한다.
- 실패 시 WAFL error toast가 표시되어야 한다.

### PDF

- 작업지시서 PDF 생성 버튼이 기존 위치에서 정상 동작해야 한다.
- 자재 발주 PDF와 작업지시서 PDF가 서로 다른 흐름으로 유지되어야 한다.
- PDF 생성 실패 시 오류 안내가 표시되어야 한다.
- PDF 생성 성공 후 첨부 목록에 연결되는 경우 기존 정책대로 표시되어야 한다.

### 첨부/디자인/메모

- 문서 첨부 업로드가 정상 동작해야 한다.
- 디자인 업로드가 정상 동작해야 한다.
- 대표 디자인 지정이 정상 동작해야 한다.
- 첨부/디자인 삭제가 정상 동작해야 한다.
- 메모 작성/수정/삭제가 정상 동작해야 한다.
- 답글 작성/수정/삭제가 정상 동작해야 한다.
- 첨부/디자인/메모 count 또는 empty state가 실제 데이터와 일치해야 한다.

## 원단·부자재 발주 QA

### 발주서 목록

- 발주서 목록이 회사 범위 기준으로 조회되어야 한다.
- 발주서 생성이 정상 동작해야 한다.
- 발주서 선택 시 중앙 상세와 우측 작업지시서 자재 선택 영역이 같은 발주서를 가리켜야 한다.
- 검색 결과 없음, 조회 실패, 로딩 상태가 WAFL 상태 UI로 표시되어야 한다.

### 발주 상세

- 발주 구분 변경이 정상 동작해야 한다.
- 공급처 선택이 정상 동작해야 한다.
- 공급처 조회 실패 후 다시 조회가 정상 동작해야 한다.
- 발주 품목 수량/단가 수정이 정상 동작해야 한다.
- 발주 품목 삭제가 정상 동작해야 한다.
- 발주 품목 empty row가 화면을 깨지 않아야 한다.

### 작업지시서 자재 할당

- 우측 패널에서 작업지시서 자재 후보가 조회되어야 한다.
- 이미 선택된 자재, 완료된 자재, 진행 중 자재는 정책에 맞게 disabled 처리되어야 한다.
- 자재 선택 후 발주 품목에 반영되어야 한다.
- 같은 회사 범위를 벗어난 작업지시서/자재가 섞이면 안 된다.

### 상태/PDF

- 발주 상태 변경이 정상 동작해야 한다.
- 자재 발주 PDF 생성이 정상 동작해야 한다.
- 작업지시서 상태와 자재 발주 완료 상태가 정책에 맞게 연결되어야 한다.
- 실패 시 WAFL error toast가 표시되어야 한다.

## 저장소/R2/휴지통 QA

### 저장소 목록

- 문서/디자인 목록 조회가 정상 동작해야 한다.
- empty/loading/error 상태가 WAFL 기준으로 표시되어야 한다.
- 미리보기/다운로드/삭제가 정상 동작해야 한다.

### 휴지통

- 삭제된 작업지시서/문서/디자인/메모가 역할별 라벨 기준으로 표시되어야 한다.
- 선택 복원/선택 삭제/전체 비우기 동작이 정상이어야 한다.
- 삭제/복원 후 storage stats가 갱신되어야 한다.
- R2 purge 흐름은 기존 정책을 유지해야 한다.

## 멤버/권한 QA

### 고객사 관리자

- 전체 작업지시서 조회 가능
- 작업지시서 생성 가능
- 담당자 변경 가능
- 검토완료/발주/검수 가능
- 멤버관리 접근 가능
- 설정 접근 가능

### 일반 멤버

- 본인 담당 작업지시서만 조회 가능
- 권한 체크에 따라 생성/검토요청/발주요청/검수 가능 여부가 달라져야 한다.
- 권한이 없는 경우 UI 제한과 서버 거부가 모두 안전해야 한다.

### 검수 권한

- 검수 단계에서만 검수 승인/반려/완료 버튼이 표시되어야 한다.
- 검수 권한이 없는 사용자는 서버에서도 거부되어야 한다.

## 서버 거부 QA

UI에서 버튼이 보이지 않아도 다음은 서버에서 반드시 거부되어야 한다.

- 다른 회사 작업지시서 조회/수정/삭제
- 다른 회사 첨부/디자인/메모 접근
- 권한 없는 상태 변경
- 권한 없는 작업지시서 생성
- 권한 없는 발주 상태 변경
- system_admin 전용 API에 일반 사용자가 접근

## 기능 추가 전 권장 순서

1. 0.19.56 QA 기준 확정
2. 0.19.57 실제 QA 결과 보정 1차
3. 0.19.58 작업지시서 저위험 기능 수정
4. 0.19.59 원단·부자재 저위험 기능 수정
5. 0.19.60 멤버/권한/환경설정 기능 수정
6. 0.19.61 시스템관리자 운영 기능 정리 착수

## 제외 범위

이번 문서는 기준 정리만 한다. 다음은 포함하지 않는다.

- DB schema 변경
- full_reset.sql 변경
- API route 분해
- R2 key 정책 변경
- PDF 생성 로직 변경
- 작업지시서 workflow 상태 추가/삭제
- 원단·부자재 발주 계산식 변경
