# 작업지시서 operation 피드백 통합 — 0.23.31

## 목적

작업지시서의 변경 진행 표시, 성공 결과, 실패 결과가 서로 다른 표시 계층을 사용하면서 발생하던 다음 문제를 정리한다.

- 진행 표시와 성공 토스트가 동시에 노출됨
- 성공 후 이전 진행 표시가 잠깐 다시 나타남
- 성공 메시지가 info tone으로 표시됨
- 작업지시서와 발주서의 loading → success/error 전환 방식이 다름
- 필드별 변경 대상 문구가 충분히 구체적이지 않음

## 적용 정책

- 작업지시서 변경 작업은 `useWorkOrderFeedback`을 통해 동일한 operation ID를 사용한다.
- 하나의 toast가 `loading → success/error`로 교체된다.
- 성공은 `success` tone, 실패는 `danger` tone을 사용한다.
- 변경 operation이 활성화된 동안 별도의 processing overlay를 만들지 않는다.
- 전체 저장, 납기일, 수량, 분류, 작업지시서명, 담당자, 재고 수량을 구분한다.

## 문구 예시

- `작업지시서 기본정보(납기일) 변경 중입니다.`
- `작업지시서 기본정보(수량) 변경 중입니다.`
- `작업지시서 기본정보(분류) 변경 중입니다.`
- `작업지시서 기본정보(담당자) 변경 중입니다.`
- `작업지시서 기본정보(재고 수량) 변경 중입니다.`
- 성공: `정보가 변경되었습니다.`
- 실패: `정보를 변경하지 못했습니다.`

## 비고

생성, 삭제, 첨부 업로드, PDF 생성처럼 정보 필드 변경과 다른 작업은 기존 고유 문구를 유지한다.
