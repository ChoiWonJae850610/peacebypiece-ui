# 0.10.69 자동 이메일 발송 검토 문서

## 목적

초대 링크/QR 기반 가입 흐름이 잡힌 뒤 자동 이메일 발송을 바로 붙일지 검토한다. 1차 운영 원칙은 여전히 링크 복사와 QR 공유이며, 이메일 자동 발송은 도메인 인증과 발송 로그 정책이 준비된 뒤 도입한다.

## 현재 유지할 1차 방식

- 시스템관리자 고객사 초대: `/system/invites`에서 초대 링크와 QR을 생성한다.
- 고객관리자 내부 멤버 초대: `/admin/members`에서 초대 링크와 QR을 생성한다.
- 사용자는 링크를 카카오톡, 문자, 이메일 등 원하는 채널로 직접 전달한다.
- raw token은 1회만 노출하고 DB에는 token_hash만 저장한다.
- 초대 링크 접속 시 서버에서 token_hash, invitation_type, status, expires_at을 다시 검증한다.

## 자동 이메일 발송 후보

### Resend

초기 SaaS 초대 메일 테스트 후보로 본다.

장점:
- Next.js 프로젝트와 연결하기 쉽다.
- 개발자 경험이 단순하다.
- 초대 메일 템플릿 테스트가 빠르다.

주의:
- 운영 도메인 인증이 필요하다.
- 발송 한도와 요금제를 확인해야 한다.
- 운영 로그와 재발송 정책은 별도로 설계해야 한다.

### AWS SES

발송량이 늘어난 뒤 장기 운영 후보로 본다.

장점:
- 대량 발송 단가가 낮은 편이다.
- 반송/불만 처리 등 운영 구성이 가능하다.
- 장기 확장성이 높다.

주의:
- 초기 설정이 상대적으로 복잡하다.
- sandbox 해제와 DNS 설정이 필요하다.
- 운영 모니터링과 실패 처리 설계가 필요하다.

### SendGrid

마케팅성 메일과 거래성 메일을 같이 키울 경우 비교 후보로 본다.

장점:
- 템플릿과 통계 기능이 많다.
- 운영 사례가 많다.
- 이메일 관리 기능이 풍부하다.

주의:
- 설정 항목이 많다.
- 요금제를 확인해야 한다.
- 국내 수신 품질은 별도 테스트가 필요하다.

## 도메인 인증 필요 사항

자동 이메일 발송 전 아래 항목을 준비한다.

1. 발송 전용 도메인 또는 서브도메인 결정
2. SPF 설정
3. DKIM 설정
4. DMARC 정책 설정
5. 반송 메일 처리 주소 결정
6. 발송자 이름 정책 결정
7. 테스트 수신함 기준 정리

초기에는 `noreply` 성격의 발송 주소와 실제 문의용 주소를 분리하는 것이 안전하다.

## 초대 이메일 템플릿

### 고객사 초대 메일

필수 변수:
- 고객사 후보명
- 초대 링크
- 만료일
- 초대한 시스템관리자 이름

메일 목적:
- 고객사 담당자가 `/invite/company/[token]`으로 이동해 가입 신청을 남기게 한다.
- 가입 신청만으로 고객사가 생성되지 않고 시스템관리자 승인 시점에 확정된다는 점을 안내한다.

### 내부 멤버 초대 메일

필수 변수:
- 회사명
- 초대 링크
- 권한 기본 묶음명
- 만료일
- 초대한 고객관리자 이름

메일 목적:
- 디자이너, 검수담당자, 재고담당자 등이 `/invite/member/[token]`으로 이동해 가입 신청을 남기게 한다.
- 승인 전에는 `/pending` 외 업무 화면 접근이 제한된다는 점을 안내한다.

## 실패/재발송 정책

자동 발송을 도입할 때는 invitations와 별도로 발송 로그를 남긴다.

필요 상태 후보:
- queued
- sent
- delivered
- bounced
- failed
- retried

재발송 정책:
- 같은 invitation token을 재사용할지, 새 token을 발급할지 결정해야 한다.
- 만료된 초대는 재발송하지 않고 새 초대를 생성한다.
- 발송 실패가 있어도 invitation 자체 상태와 email delivery 상태는 분리한다.

## SMS/카카오 알림톡 후순위 판단

SMS는 건당 비용과 발신번호 등록이 필요하다. 카카오 알림톡은 비즈니스 채널과 템플릿 심사가 필요하다. 따라서 1차 초대 기능에서는 제외하고, 링크 복사와 QR 공유가 충분히 동작한 뒤 검토한다.

## 0.10.69 적용 내용

- 자동 이메일 발송 검토 문서를 추가했다.
- 초대 발송 채널 정책을 `lib/invitations/invitationDeliveryPolicy.ts`로 분리했다.
- 현재 기본 발송 채널은 `manual_link`로 유지한다.
- 이메일 provider 후보와 초대 이메일 템플릿 정책을 코드 상수로 정리했다.
- 실제 이메일 발송 API, provider SDK, package.json 변경은 하지 않았다.

## 후속 작업 기준

자동 이메일 발송을 실제 도입하려면 다음 순서가 적절하다.

1. 발송 provider 확정
2. 도메인 인증 설정
3. email_delivery_logs 테이블 설계
4. 초대 생성 API와 이메일 발송 분리
5. 발송 실패/재발송 정책 적용
6. 시스템관리자 발송 로그 조회 화면 추가
