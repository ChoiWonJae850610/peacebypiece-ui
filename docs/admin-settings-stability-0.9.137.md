# 관리자 설정 화면 안정화 점검 — 0.9.137

## 목적

0.9.77 복구 라인의 `/admin/settings` 본 기능 UI를 유지하면서 설정 저장 중복 실행 위험을 줄이고, 향후 고객사별 파일 보관 정책과 시스템관리자 R2 purge 기능을 연결할 위치를 고정한다.

## 현재 화면 역할

- 회사 UI 설정: 테마 색상, 언어 설정을 저장한다.
- 정책 관리: 알림 정책, 로그 이벤트, 저장 정책을 관리한다.
- 기준 관리: 생산품 유형, 단위 표준, 외주 공정 유형을 관리한다.
- 사용자 접근 preview: 현재 사용자 접근 상태를 확인한다.

## 이번 안정화 범위

### 회사 UI 설정

- 테마/언어 저장 중에는 추가 클릭을 무시한다.
- 저장 중에는 테마/언어 버튼을 비활성화한다.
- 기존 자동 저장 흐름과 `runSaveCompanySettingsFlow`는 유지한다.

### 파일 정책 모달

- 저장 또는 로딩 중에는 기본값 복원, 저장, 토글, 보관일, 용량 한도, 경고 기준 입력을 비활성화한다.
- `trashRetentionDays`, `storageLimitGb`, `warningThresholdPercent` normalize 흐름은 유지한다.
- 파일 보관 기간은 향후 `purge_due_at` 계산의 기준값으로 사용한다.

### 알림 정책 모달

- 저장 또는 로딩 중에는 기본값 복원, 저장, 알림 토글을 비활성화한다.
- 삭제 결과 알림(`purgeResultEnabled`)은 향후 시스템관리자 R2 purge 완료/실패 알림과 연결 가능한 정책값으로 유지한다.

### 외주 공정 유형 모달

- 저장 중에는 저장, 기본값 복원, 신규 공정 추가, 이동 버튼, 선택 변경을 비활성화한다.
- 저장 성공 전에는 모달을 닫지 않는다.
- 저장 실패 시 모달을 유지하고 에러 메시지를 표시한다.

## 향후 R2 purge 연결 기준

관리자 설정 화면의 저장 정책은 R2 실제 삭제 기능을 직접 실행하지 않는다. 이 화면은 고객사별 보관 정책값만 관리한다.

향후 시스템관리자 purge 흐름은 아래 순서로 분리한다.

1. 고객사 관리자 또는 작업지시서에서 파일 삭제
2. DB metadata에 삭제 상태와 `deleted_at` 기록
3. 고객사 파일 정책의 `trashRetentionDays` 기준으로 `purge_due_at` 계산
4. 시스템관리자 화면에서 삭제 예정 도래 파일 목록 표시
5. 시스템관리자가 목록/모달에서 확인 후 Worker 기반 R2 실제 삭제 실행
6. DB에 purge 완료/실패 상태 기록

## 이번 버전에서 하지 않은 것

- DB schema 변경
- `purge_due_at` 컬럼 추가
- 시스템관리자 R2 purge 화면 추가
- 자동 삭제/cron/scheduled trigger 추가
- read-only/skeleton route 재도입
- package 의존성 변경

## QA 체크리스트

1. `/admin/settings` 진입 시 runtime error가 없어야 한다.
2. 테마 색상을 빠르게 여러 번 클릭해도 저장 요청이 중복 폭주하지 않아야 한다.
3. 언어 버튼을 빠르게 여러 번 클릭해도 저장 중에는 버튼이 비활성화되어야 한다.
4. 저장 정책 모달에서 저장 중 보관일/용량/토글 값이 추가 변경되지 않아야 한다.
5. 알림 정책 모달에서 저장 중 토글이 추가 변경되지 않아야 한다.
6. 외주 공정 유형 모달에서 저장 중 추가/이동/기본값 복원/저장이 비활성화되어야 한다.
7. 저장 실패 시 모달이 닫히지 않고 에러 메시지가 남아야 한다.
8. 작업지시서 첨부/메모/R2 업로드·삭제 흐름은 기존처럼 동작해야 한다.
