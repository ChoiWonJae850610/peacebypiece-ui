# 0.9.139 — 30일 고정 휴지통 정책 및 R2 purge 설계 기준

## 목적

0.9.139에서는 고객사별 삭제 보관기간 설정을 제공하지 않고, 전 고객 공통 30일 휴지통 보관 정책으로 고정한다.
고객관리자는 파일 삭제 정책을 선택하지 않으며, 삭제된 파일은 항상 휴지통으로 이동한다.

## 확정 정책

1. 고객사 관리자는 삭제 방식과 보관기간을 설정하지 않는다.
2. 모든 첨부파일 삭제는 휴지통 이동으로 처리한다.
3. 휴지통 파일은 삭제일로부터 30일 동안 복원할 수 있다.
4. 30일이 지난 휴지통 파일은 영구 삭제 후보가 된다.
5. 영구 삭제 후보가 되어도 R2 물리 삭제는 즉시 실행하지 않는다.
6. R2 실제 삭제는 시스템관리자 purge 화면 또는 purge job에서 수행한다.
7. R2 삭제 시 원본 `storage_key`와 썸네일 `thumbnail_key`를 함께 삭제한다.
8. 삭제 완료 후 DB에는 purge 완료 상태를 기록한다.

## 고객관리자 화면 기준

고객관리자 저장정책 모달에서는 다음 항목을 제거한다.

- 삭제 방식 선택
- 즉시삭제 선택
- 휴지통 보관기간 선택
- 1일 / 5일 / 15일 / 30일 선택 버튼

대신 다음 안내만 표시한다.

> 삭제된 파일은 휴지통으로 이동하며 30일 동안 복원할 수 있습니다. 30일이 지나면 영구 삭제 대상이 됩니다.

## 내부 계산 기준

```text
deleted_at + 30일 = purge 후보 가능 시점
```

기존 DB의 `soft_delete_enabled`, `trash_retention_days` 값이 남아 있어도 런타임 정책은 다음처럼 고정한다.

```text
softDeleteEnabled = true
trashRetentionDays = 30
```

## R2 삭제 기준

휴지통 이동 시에는 R2 객체를 삭제하지 않는다.

```text
사용자 삭제
→ attachments deleted 상태 기록
→ attachment_trash_items 생성
→ R2 원본/썸네일 유지
→ 30일 경과
→ 시스템관리자 purge 후보
→ Worker로 R2 원본 + 썸네일 삭제
→ DB purge 완료 기록
```

## 0.9.139에서 실제 반영한 범위

- 고객관리자 저장정책 모달에서 삭제 방식/보관기간 선택 UI를 제거했다.
- 기본 파일 정책을 30일 고정 휴지통 정책으로 변경했다.
- settings normalize 로직에서 기존 DB 값과 무관하게 `softDeleteEnabled=true`, `trashRetentionDays=30`으로 고정했다.
- 첨부 삭제 API가 항상 soft delete를 수행하도록 정리했다.
- R2 직접 삭제 방식은 첨부 삭제 API에서 사용하지 않는다.
