# 정책 강제 재동의/차단 UX 설계 0.20.00

## 1. 목적

WAFL 서비스의 약관, 개인정보처리방침, 요금·환불·저장소·데이터 보관 정책 중 중요한 변경이 발생했을 때 사용자가 변경 내용을 확인하고 다시 동의해야 하는 기준을 정리한다.

이번 버전은 설계 문서 단계다. DB schema, API route, 화면 차단 로직, R2/첨부/메모/휴지통/purge 흐름은 변경하지 않는다.

## 2. 적용 원칙

정책 재동의는 단순 공지와 구분한다. 모든 정책 버전 변경이 업무 차단으로 이어지면 운영 부담과 사용자 혼란이 커지므로, 실제 차단은 중요 변경에만 적용한다.

재동의 필요 여부는 정책 버전 단위에서 판단한다. 기준 필드는 `policy_versions.requires_reagreement`를 사용한다.

재동의가 필요한 정책은 다음 조건을 모두 만족해야 한다.

- 현재 활성 정책 버전이다.
- 사용자에게 필수로 적용되는 정책이다.
- `requires_reagreement = true`이다.
- 현재 사용자의 `policy_agreements`에 해당 정책 버전에 대한 동의 기록이 없다.

## 3. 중요 변경 기준

다음 변경은 재동의 필요 대상으로 본다.

- 개인정보 수집·이용 목적의 실질 변경
- 개인정보 보관 기간 또는 파기 기준의 실질 변경
- 유료 요금제, 무료체험, 환불, 해지, 결제 실패 처리 기준의 실질 변경
- 저장공간 제한, 업로드 차단, 데이터 보관·삭제 기준의 실질 변경
- 고객사·멤버의 책임 범위가 확대되는 약관 변경
- 법령 또는 외부 서비스 정책 변경으로 인해 명시적 재동의가 필요한 경우

다음 변경은 기본적으로 재동의 없이 고지 대상으로 본다.

- 오탈자 수정
- 고객지원 이메일, 회사명 표기, 화면 문구 등 비실질 변경
- 조항 번호 변경
- 기존 의미를 바꾸지 않는 설명 보강

## 4. 적용 사용자 범위

### 고객사 관리자

고객사 관리자는 회사 설정, 멤버관리, 정책 확인, 요금제·저장공간 관리에 접근하는 주체이므로 필수 정책 미동의 시 업무 화면 진입을 차단한다.

차단 대상 예시는 다음과 같다.

- 업무홈
- 작업지시서
- 협력업체관리
- 멤버관리
- 환경설정의 업무성 영역
- 저장소관리
- 통계정보

### 일반 멤버

일반 멤버도 작업지시서, 첨부, 메모, 발주 흐름에 접근할 수 있으므로 필수 정책 미동의 시 업무 화면 진입을 차단한다.

일반 멤버에게는 고객사 관리자용 요금제·회사설정 설명을 노출하지 않고, 본인 계정 기준으로 필요한 정책 재동의만 안내한다.

### 시스템관리자

시스템관리자 운영 화면의 차단 여부는 별도 정책으로 분리한다. 시스템관리자는 고객사 운영을 지원해야 하므로 고객사 정책 미동의 상태와 동일하게 묶지 않는다. 다만 시스템관리자 자체 운영약관 또는 내부 보안정책 동의가 필요한 경우에는 별도 플래그로 설계한다.

## 5. 차단 예외 화면

재동의가 필요한 상태에서도 다음 기능은 허용한다.

- 정책/약관 열람 화면
- 재동의 제출 액션
- 로그아웃
- 고객지원 안내
- 계정 식별에 필요한 최소 세션 확인

다음 기능은 차단한다.

- 파일 업로드
- 작업지시서 생성/수정/삭제
- 발주/검수 상태 변경
- 멤버 초대·승인·탈퇴 처리
- 회사 정보 변경
- 저장소 파일 열람·복원·영구삭제

## 6. 사용자 흐름

### 정상 흐름

1. 사용자가 로그인한다.
2. 서버가 현재 사용자에게 재동의가 필요한 필수 정책 목록을 조회한다.
3. 재동의 필요 정책이 없으면 업무 화면을 정상 표시한다.
4. 재동의 필요 정책이 있으면 업무 화면 대신 재동의 안내 화면을 표시한다.
5. 사용자는 변경된 정책 문서를 열람한다.
6. 사용자가 필수 정책에 모두 동의하면 `policy_agreements`에 동의 기록을 저장한다.
7. 저장 완료 후 원래 진입하려던 업무 화면으로 이동한다.

### 재동의 거부/보류 흐름

사용자가 동의를 보류하면 업무 화면은 계속 차단한다. 이 상태에서 허용되는 선택지는 정책 열람, 고객지원 확인, 로그아웃이다.

### 복수 정책 재동의 흐름

동시에 여러 필수 정책의 재동의가 필요한 경우 한 화면에서 목록을 보여준다. 각 정책은 제목, 변경일, 시행일, 필수 여부, 열람 버튼을 표시한다.

전체 동의 버튼은 모든 필수 정책을 열람 가능하게 제공한 뒤 활성화한다. 실제 구현에서는 각 정책별 체크박스 또는 전체 확인 체크박스를 둘 수 있다.

## 7. 화면 구조

재동의 차단 화면은 다음 정보를 포함한다.

- 제목: 정책 재동의가 필요합니다
- 설명: 중요한 정책 변경으로 인해 계속 이용하려면 재동의가 필요하다는 안내
- 정책 목록
  - 정책명
  - 버전
  - 시행일
  - 필수/선택 구분
  - 보기 버튼
- 전체 동의 버튼
- 로그아웃 버튼
- 고객지원 안내

업무 화면 위에 모달로만 띄우는 방식은 피한다. 업무 데이터가 뒤에 렌더링되면 차단 의미가 흐려지고 권한/데이터 노출 위험이 생기므로, 재동의 필요 상태에서는 업무 화면 대신 전용 차단 화면을 렌더링하는 방향이 안전하다.

## 8. 라우팅 차단 기준

후속 구현에서는 업무 라우트 진입 시 다음 순서로 확인한다.

1. 세션 존재 여부
2. 사용자/회사 상태
3. 필수 정책 재동의 필요 여부
4. 화면별 권한
5. 실제 업무 데이터 조회

정책 미동의 상태에서는 업무 데이터 조회 전에 차단해야 한다. 특히 작업지시서, 회사 파일, 첨부, 메모, 저장소 파일 목록은 재동의 확인 전 조회하지 않는 것을 기준으로 한다.

## 9. DB/API 후속 구현 기준

0.20.01에서 검토할 API 후보는 다음과 같다.

- `GET /api/policies/reagreement-required`
  - 현재 사용자에게 재동의가 필요한 정책 목록 조회
- `POST /api/policies/reagreement`
  - 필수 정책 재동의 저장
- `GET /api/admin/policy-status` 또는 기존 정책 상태 API 확장
  - 고객사 관리자 화면에서 정책 동의 상태 표시

DB는 기존 `policy_versions.requires_reagreement`와 `policy_agreements` 구조를 우선 활용한다. 새 테이블은 필요성이 명확할 때만 추가한다.

## 10. 자동테스트 기준

0.20.01 이후 자동테스트는 다음 항목을 포함한다.

### DB/API smoke

- 재동의 필요 정책 조회
- 이미 동의한 정책은 재동의 목록에서 제외
- 재동의 저장 후 조회 결과가 비어야 함
- rollback transaction 안에서 test data가 남지 않아야 함

### Playwright E2E

- 재동의 필요 사용자로 업무 화면 진입 시 차단 화면 표시
- 정책 문서 보기 가능
- 로그아웃 가능
- 재동의 완료 후 업무 화면 진입 가능
- 재동의가 필요 없는 사용자는 기존 업무 화면 smoke 유지

## 11. 버전 분리

- 0.20.00: 설계 문서만 추가
- 0.20.01: 정책 재동의 필요 상태 DB/API 1차
- 0.20.02: 정책 재동의 안내 UI 1차
