# 92. 작업지시서 회귀 테스트 체크리스트

## 목적

0.15.73.3부터 0.15.79까지 작업지시서 생성, 권한, 상세 hydration, 생산정보 draft, 검토요청 흐름에서 발생한 회귀를 기준으로 이후 패치마다 확인할 최소 테스트 범위를 정의한다.

이 문서는 기능 설계 문서가 아니라 회귀 방지용 테스트 기준이다. 테스트는 고객사 관리자와 디자이너 계정을 모두 사용한다.

## 전제 조건

- DB는 현재 개발 스키마 기준으로 준비되어 있어야 한다.
- 고객사 관리자 계정은 해당 회사의 전체 작업지시서를 조회할 수 있어야 한다.
- 디자이너 계정은 멤버관리에서 작업지시서 관리 권한을 켜고 끌 수 있어야 한다.
- 작업지시서 repository mode는 DB 모드 기준이다.
- 메모, 첨부, R2, 휴지통, purge 흐름은 별도 회귀가 없는 한 이 체크리스트에서 직접 수정하지 않는다.

## 1. 권한 / 생성 버튼

### 1-1. 디자이너 생성 권한 ON

1. 고객사 관리자로 로그인한다.
2. 멤버관리에서 디자이너 계정의 작업지시서 관리 권한을 켠다.
3. 디자이너 계정으로 로그인한다.
4. `/api/auth/me` 응답의 `permissionCodes`에 `workorder.create`가 포함되는지 확인한다.
5. `/worker` 화면에 진입한다.
6. 작업지시서 생성 버튼이 표시되는지 확인한다.
7. 생성 버튼 클릭 시 작업지시서가 생성되는지 확인한다.
8. `WORKORDER_PERMISSION_REQUIRED` 오류가 발생하지 않는지 확인한다.

기대 결과:

- `workorder.create`가 있으면 생성 버튼이 보인다.
- 생성 버튼 표시 조건과 실제 생성 액션 권한 조건이 일치한다.

### 1-2. 디자이너 생성 권한 OFF

1. 고객사 관리자로 로그인한다.
2. 멤버관리에서 디자이너 계정의 작업지시서 관리 권한을 끈다.
3. 디자이너 계정으로 다시 로그인하거나 새로고침한다.
4. `/worker` 화면에 진입한다.

기대 결과:

- 작업지시서 메뉴와 목록 조회는 가능하다.
- 생성 버튼은 표시되지 않거나 생성 불가 상태여야 한다.
- 서버 오류가 아니라 UI 정책에 맞게 막혀야 한다.

## 2. 작업지시서 생성 / 목록 / 상세 hydration

### 2-1. 생성 직후 상세 표시

1. 작업지시서 생성 버튼을 클릭한다.
2. 왼쪽 목록에 새 작업지시서가 표시되는지 확인한다.
3. 생성된 작업지시서가 자동 선택되는지 확인한다.
4. 상세 화면이 `불러오는 중` 상태에서 멈추지 않는지 확인한다.

기대 결과:

- 생성 응답은 상세 snapshot으로 취급된다.
- 생성 직후 상세 화면이 정상 표시된다.

### 2-2. 새로고침 후 목록 유지

1. 작업지시서를 하나 생성한다.
2. 브라우저를 새로고침한다.
3. `/worker` 목록에서 방금 생성한 작업지시서가 보이는지 확인한다.
4. 고객사 관리자와 디자이너 계정에서 각각 확인한다.

기대 결과:

- 고객사 관리자는 회사 범위의 작업지시서를 볼 수 있다.
- 디자이너는 본인 담당 범위 정책에 맞는 작업지시서를 볼 수 있다.
- summary query에서 company scope 오류가 발생하지 않는다.

## 3. 제목 / 담당자 즉시 저장과 draft 보존

### 3-1. 제목 변경 시 미저장 생산정보 보존

1. 새 작업지시서를 생성한다.
2. 발주정보의 공장, 수량, 공임, 날짜를 입력한다.
3. 원단정보 또는 외주정보 일부 값을 입력한다.
4. 아직 검토요청이나 workflow action을 누르지 않은 상태에서 작업지시서 제목을 변경한다.

기대 결과:

- 화면에 입력한 발주정보, 원단정보, 외주정보가 사라지지 않는다.
- 제목 변경 저장 결과가 draft-only 필드를 덮지 않는다.

### 3-2. 담당자 변경 시 미저장 생산정보 보존

1. 새 작업지시서에 발주정보와 원단정보를 입력한다.
2. 담당자를 변경한다.

기대 결과:

- 담당자 변경 후에도 입력 중인 발주정보, 원단정보, 외주정보가 유지된다.

## 4. 발주정보 / 생산정보 draft와 저장 정책

### 4-1. 입력 중 자동 저장 금지

1. 발주정보 수량을 여러 번 입력한다.
2. 공임비, 손실비, 날짜, 공장 값을 변경한다.
3. Network 탭에서 저장 API 요청이 입력마다 발생하는지 확인한다.

기대 결과:

- 숫자 하나 입력할 때마다 DB 저장 요청이 발생하지 않는다.
- 공장, 날짜 변경도 즉시 저장되지 않는다.

### 4-2. 발주추가 row 추가

1. 발주정보 영역에서 발주추가를 클릭한다.
2. row가 화면에 추가되는지 확인한다.
3. 추가한 row에 공장, 수량, 공임, 날짜를 입력한다.

기대 결과:

- row 추가가 정상 동작한다.
- row 추가 자체가 불필요한 DB 저장으로 이어지지 않는다.

### 4-3. workflow action 시 현재 draft 기준 저장/검증

1. 발주정보에 공장, 수량, 공임, 날짜를 입력한다.
2. 원단정보와 외주정보 일부를 입력한다.
3. 검토요청을 클릭한다.

기대 결과:

- 현재 화면 draft가 workflow 검증 기준으로 사용된다.
- 공장을 입력했는데도 `공장을 입력하세요` 오류가 나오지 않는다.
- workflow action 전에 필요한 생산정보 저장이 누락되지 않는다.

## 5. 임시 저장 버튼 제거 확인

1. 작업지시서 상세의 진행 단계 영역을 확인한다.
2. `임시 저장` 버튼이나 관련 힌트 문구가 남아 있는지 확인한다.

기대 결과:

- `임시 저장` 버튼이 표시되지 않는다.
- 발주정보/생산정보 저장 정책이 버튼명 때문에 오해되지 않는다.
- 메모 저장과 첨부 업로드 버튼은 정상 유지된다.

## 6. 메모 / 첨부 회귀 확인

### 6-1. 메모

1. 메모를 작성하고 저장한다.
2. 새로고침한다.
3. 저장한 메모가 유지되는지 확인한다.

기대 결과:

- 메모 저장은 명시 저장 액션으로 동작한다.
- 작업지시서 draft 정리와 충돌하지 않는다.

### 6-2. 첨부

1. 디자인 첨부 또는 일반 첨부를 업로드한다.
2. 대표 디자인 지정이 필요한 경우 지정한다.
3. 새로고침한다.

기대 결과:

- 첨부가 정상 유지된다.
- R2 실제 삭제/purge 흐름은 변경되지 않는다.

## 7. 최소 회귀 테스트 순서

작업지시서 관련 패치 후에는 아래 순서로 먼저 확인한다.

1. `npm run build`
2. 고객사 관리자 `/worker` 목록 표시
3. 디자이너 `/worker` 목록 표시
4. 디자이너 생성 권한 ON/OFF
5. 작업지시서 생성 직후 상세 표시
6. 새로고침 후 목록 유지
7. 제목 변경 시 draft 보존
8. 발주정보 입력 중 자동 저장 없음
9. 검토요청 시 현재 draft 기준 검증
10. 메모 저장
11. 첨부 저장
