# 정책 미동의 업무 접근 차단 1차 (0.20.03)

## 목적

`0.20.01`에서 추가한 정책 재동의 필요 상태 API와 `0.20.02`에서 추가한 재동의 안내 UI를 업무 화면 접근 차단 UX에 연결한다.

이번 버전은 고객사 관리자와 일반 멤버가 중요 정책 변경사항에 재동의하지 않은 상태에서 업무 화면을 계속 사용하는 것을 막는 1차 구현이다.

## 적용 범위

- 적용 대상: `/workspace/*` 업무 영역
- 예외 허용 경로: `/workspace/legal`
- 적용 사용자: 고객사 관리자, 일반 멤버
- 제외 사용자: 시스템관리자 운영 영역(`/system/*`)

## 차단 조건

`GET /api/policies/reagreement` 응답의 `status.hasPendingReagreement`가 `true`이면 업무 화면을 차단한다.

차단 화면은 다음 정보를 제공한다.

- 정책 재동의 필요 안내
- 남은 재동의 건수
- 약관·정책 화면 이동 버튼
- 고객지원 문의 링크
- 상태 다시 확인 버튼
- 로그아웃 버튼

## 허용 작업

재동의 미완료 상태에서도 다음 작업은 허용한다.

- 약관·정책 화면 열람
- 필수 정책 전체 재동의 저장
- 로그아웃
- 고객지원 문의

## 차단 작업

재동의 미완료 상태에서는 다음과 같은 업무 화면 사용을 제한한다.

- 작업지시서
- 원단·부자재
- 협력업체 관리
- 통계정보
- 기준정보
- 저장소 관리
- 환경설정
- 기타 `/workspace/legal`이 아닌 업무 영역

## 구현 메모

- `components/policies/PolicyReagreementAccessGate.tsx`를 추가했다.
- `app/(workspace)/workspace/layout.tsx`에서 업무 영역 children을 `PolicyReagreementAccessGate`로 감싼다.
- 차단 게이트는 클라이언트에서 `/api/policies/reagreement`를 조회한다.
- 조회 실패 또는 세션 미확정 상태에서는 기존 화면을 막지 않는다.
- `/workspace/legal`은 재동의 완료를 위한 예외 경로로 유지한다.

## 자동테스트 기준

`tests/e2e/workspace-policy-settings.spec.mjs`에 다음 조건을 추가했다.

- 재동의 대기 상태 mock일 때 `/workspace/settings` 진입 시 차단 안내가 표시되어야 한다.
- 차단 안내에는 `약관·정책 확인하기`, `로그아웃`, `고객지원 문의`가 표시되어야 한다.
- 기존 설정 화면 smoke는 재동의 완료 상태 mock으로 계속 통과해야 한다.

## 후속 작업

- `0.20.04`부터 요금제/무료체험/저장공간 운영 데이터 설계로 이동한다.
- 서버 레벨에서 업무 데이터 조회 전 차단이 필요한 API는 별도 access guard 보강 대상으로 분리할 수 있다.
