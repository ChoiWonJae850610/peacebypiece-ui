# 0.9.224391 — 관리자/멤버 IA 확정 설계

## 목적

고객관리자 좌측 패널 제거, 카드형 홈 전환, 작업지시서 업무 화면의 홈 복귀, 멤버별 메인화면, 권한 기반 카드 노출 원칙을 확정한다.

이번 버전은 설계 문서 중심이다. 실제 권한 DB, 멤버 초대, 권한 관리 UI, 라우트 보호, API 권한 검사는 구현하지 않는다.

## 확정 방향

### 1. 고객관리자 좌측 패널 완전 삭제

고객관리자 화면에서는 좌측 패널을 완전히 삭제하는 방향으로 간다.

고객관리자는 관리자 도구를 계속 탐색하는 사용자라기보다 필요한 기능으로 바로 진입하는 사용자로 본다. 따라서 `/admin`은 메뉴 중심 화면이 아니라 카드형 홈으로 전환한다.

### 2. 고객관리자 기본 관리 카드

고객관리자 홈의 기본 관리 카드는 아래 항목으로 제한한다.

1. 협력업체관리
2. 저장소관리
3. 통계정보
4. 환경설정
5. 멤버관리 후보

작업지시서는 이 관리 카드 목록에 포함하지 않는다.

### 3. 작업지시서 업무 화면 진입

작업지시서는 관리 메뉴가 아니라 업무 실행 화면으로 본다.

다만 고객관리자와 멤버가 작업지시서에 진입해야 하는 상황이 있으므로 `/admin` 또는 역할별 홈에는 별도 업무 진입 버튼을 둔다.

권장 표시명:

- 작업지시서 업무 화면 열기

이 버튼은 기본 관리 카드와 시각적으로 분리한다.

### 4. 작업지시서 화면 홈 버튼

작업지시서 화면에는 홈 버튼을 제공한다.

홈 버튼은 고정으로 `/admin`에 보내지 않고, 현재 사용자의 역할 또는 권한 기준 메인화면으로 되돌린다.

초기 설계 기준:

1. 고객관리자 → `/admin`
2. 시스템관리자 → `/system`
3. 일반 멤버 → `/workspace` 후보

`/workspace`가 아직 구현되지 않은 상태에서는 문서상 후보로 유지하고, 실제 구현 시 현재 라우팅 구조에 맞는 최소 이동만 적용한다.

### 5. 관리자뿐 아니라 멤버도 메인화면 필요

장기적으로 모든 일반 사용자는 각자의 메인화면이 필요하다.

예상 사용자 유형:

1. 고객관리자
2. 디자이너
3. 검수자
4. 생산/발주 담당자
5. 재고 담당자
6. 기타 멤버

이 메인화면은 역할 이름만으로 고정하지 않는다. 실제 표시 카드는 권한 기준으로 결정한다.

## 권한 기반 카드 원칙

### 기본 원칙

메인화면 카드는 역할 이름이 아니라 권한 코드 기준으로 노출한다.

역할은 기본 권한 묶음이다. 실제 화면 노출과 기능 접근은 permission code 기준으로 판단한다.

예:

| 권한 코드 | 표시 카드 | 후보 경로 |
| --- | --- | --- |
| `workorder.access` | 작업지시서 업무 화면 | `/worker` |
| `partner.manage` | 협력업체관리 | `/admin/partners` |
| `storage.manage` | 저장소관리 | `/admin/files` |
| `stats.view` | 통계정보 | `/admin/dashboard` |
| `settings.organization.manage` | 환경설정 | `/admin/settings` |
| `member.manage` | 멤버관리 | `/admin/members` 후보 |
| `standard_unit.manage` | 단위표준 | 추후 확정 |
| `outsourcing_process.manage` | 외주공정 | 추후 확정 |
| `product_type.manage` | 생산품유형 | 추후 확정 |

### 초기 구현 기준

당장 완성형 권한 시스템을 구현하지 않는다.

초기 구현은 다음 방식을 권장한다.

1. 카드 registry에 permission code 필드를 미리 둔다.
2. 현재는 customer_admin 기준으로 필요한 카드를 표시한다.
3. 권한 DB와 권한 관리 UI는 이후 버전에서 연결한다.
4. 문자열 직접 비교가 필요한 부분은 상수화된 permission code를 사용한다.
5. 카드 표시 로직은 TSX에 흩뿌리지 않고 selector 또는 registry 유틸로 분리한다.

## 설정 구조 분리

환경설정은 두 층으로 나눈다.

### 개인 설정

현재 로그인한 사용자의 선호 설정이다.

후보 경로:

- `/me/settings`

후보 항목:

1. 언어
2. 색상/테마
3. 화면 밀도
4. 알림 선호
5. 기본 진입 화면

### 조직 설정

회사 또는 고객 조직 단위의 운영 설정이다.

후보 경로:

- `/admin/settings`

후보 항목:

1. 저장소 정책
2. 파일 업로드 정책
3. 협력업체 정책
4. 권한 정책
5. 조직 알림 정책
6. 요금제/용량 정책 후보

## 라우트 방향

### 단기

1. `/admin`은 고객관리자 카드형 홈으로 전환한다.
2. `/admin/dashboard`는 통계정보 화면으로 유지한다.
3. `/admin/files`는 저장소관리 화면으로 유지한다.
4. `/admin/settings`는 조직 설정 화면으로 유지한다.
5. `/worker`는 작업지시서 업무 화면으로 유지한다.
6. 시스템관리자 `/system` 구조는 좌측 패널을 유지한다.

### 중기

1. `/workspace` 공통 홈을 도입한다.
2. 고객관리자와 일반 멤버 모두 권한 기반 카드 홈을 사용할 수 있도록 한다.
3. `/admin`은 고객관리자 전용 조직 관리성 경로로 축소하거나 `/workspace`와 역할을 재정의한다.

### 장기

1. 로그인 후 role/permission 기준으로 공통 홈에 진입한다.
2. 카드 노출은 permission registry에서 자동 구성한다.
3. 화면 접근과 API 접근 권한 검사를 일치시킨다.

## 이번 버전에서 하지 않는 일

1. 실제 고객관리자 좌측 패널 삭제 구현
2. `/admin` 카드형 홈 UI 구현
3. `/workspace` 라우트 추가
4. 작업지시서 홈 버튼 구현
5. roles/permissions DB 추가
6. 멤버 초대 시스템 구현
7. 권한 관리 UI 구현
8. API 권한 체크 구현
9. package.json/package-lock.json 변경
10. DB schema 변경

## 다음 작업 권장 순서

### 0.9.224392 — 고객관리자 카드형 홈 1차 구현

목표:
고객관리자 좌측 패널을 제거하고 `/admin`을 카드형 홈으로 전환한다.

주의:
실제 권한 DB는 붙이지 않는다. 카드 registry에는 permission code만 미리 포함한다.

### 0.9.224393 — 작업지시서 홈 버튼 추가

목표:
작업지시서 업무 화면에서 사용자의 메인화면으로 돌아가는 홈 버튼을 추가한다.

주의:
`/workspace`가 없으면 고객관리자 기준 `/admin` 복귀를 우선 적용하고, 일반 멤버 홈은 후속 설계로 남긴다.

### 0.9.224394 — 개인 설정/조직 설정 분리 설계

목표:
언어, 색상/테마, 알림 선호는 개인 설정으로 분리하고, 저장소/파일/권한 정책은 조직 설정으로 유지한다.

### 0.9.224395 — 멤버관리/권한관리 설계

목표:
멤버 초대, 역할, 권한 부여, 권한 기반 카드 노출을 실제 DB와 UI로 연결할 설계를 정리한다.

## 완료 판단

아래 기준을 만족하면 0.9.224391 설계를 다음 구현의 기준으로 삼는다.

1. 고객관리자 좌측 패널은 완전 삭제 방향으로 확정한다.
2. 고객관리자 기본 관리 카드는 협력업체관리, 저장소관리, 통계정보, 환경설정, 멤버관리 후보로 제한한다.
3. 작업지시서는 관리 카드가 아니라 업무 실행 화면으로 분리한다.
4. 작업지시서 화면에는 홈 복귀 버튼이 필요하다고 확정한다.
5. 관리자뿐 아니라 일반 멤버도 메인화면이 필요하다고 확정한다.
6. 메인화면 카드는 역할 이름이 아니라 permission code 기준으로 확장 가능해야 한다.
