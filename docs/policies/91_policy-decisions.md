# Policy Decisions

## 문서 상태
- Policy Version: P-0.91.34
- Based App Version: 0.15.73
- 문서 성격: 정책 결정사항 및 미결정사항 관리 문서
- 적용 범위: 개인정보처리방침, 이용약관, 결제정책, 저장소 보관정책의 공통 기준

이 문서는 정책 초안 작성 중 확정된 운영 기준, 아직 확정하지 않은 기준, 그리고 해당 정책 때문에 필요한 개발 항목을 분리해서 관리한다.

---

## 1. 확정한 정책 결정사항

### 1.1 고객사 승인 요청 시 사업자등록증 파일 필수

#### 결정
고객사 관리자가 승인 요청을 할 때 사업자등록증 파일 제출을 필수로 한다.

#### 반영 문서
- docs/policies/01_privacy-policy-draft.md
  - 2.3 고객사 회사 정보
  - 13. 내부 검토 필요 사항

#### 개발 필요
- 승인 요청 화면에서 사업자등록증 파일 필수 처리
- 파일 업로드 전 임시 보관 또는 승인 요청 시 일괄 제출 방식 유지
- 사업자등록증 파일의 저장 위치와 접근 권한 정의
- 승인 전 임시 데이터/파일의 만료 또는 폐기 기준 필요

---

### 1.2 회사 로고/이미지는 선택 항목

#### 결정
회사 로고 또는 회사 이미지는 필수가 아니라 선택 항목으로 둔다.

#### 반영 문서
- docs/policies/01_privacy-policy-draft.md
  - 2.3 고객사 회사 정보

#### 개발 필요
- 회사 로고/이미지 미등록 상태 처리
- 기본 placeholder 표시
- 나중에 회사 설정에서 추가/삭제 가능하도록 설계

---

### 1.3 대표자명은 수집하되 검증 가능성을 남긴다

#### 결정
대표자명은 고객사 회사 정보로 수집한다. 가능하면 사업자등록정보와 대표자명이 일치하는지 확인하는 로직을 추후 도입한다.

#### 반영 문서
- docs/policies/01_privacy-policy-draft.md
  - 2.3 고객사 회사 정보
  - 13. 내부 검토 필요 사항
- docs/policies/90_policy-feature-gap.md
  - 사업자등록정보 진위확인 gap

#### 개발 필요
- 대표자명 입력 필드
- 사업자등록번호 입력 필드
- 개업일 입력 필요 여부 검토
- 국세청 사업자등록정보 진위확인 API 연동 검토
- 진위확인 실패 시 승인 요청 제한 또는 관리자 검토 전환 정책 필요

#### 주의
사업자등록번호만으로는 대표자명 일치 여부까지 확인하기 어렵다. 대표자명 일치 검증을 하려면 일반적으로 사업자등록번호, 대표자명, 개업일 등 진위확인 API가 요구하는 항목을 함께 받아야 할 수 있다.

---

### 1.4 결제 방식은 카드결제 중심으로 유지

#### 결정
초기뿐 아니라 앞으로도 결제 방식은 카드결제 중심으로 운영한다. 계좌이체/무통장입금은 기본 운영 방식에서 제외한다.

#### 반영 문서
- docs/policies/01_privacy-policy-draft.md
  - 2.5 결제 관련 정보
- 향후 작성 예정:
  - docs/policies/03_billing-refund-policy-draft.md

#### 개발 필요
- 카드 등록 화면
- PG customerKey/billingKey 저장 구조
- 결제수단 상태 관리
- 결제 실패 상태 관리
- 카드 변경 기능
- 영수증 URL 관리
- 세금계산서 요청 여부 및 발행 상태 관리

---

### 1.5 카드 원문 정보는 저장하지 않는다

#### 결정
전체 카드번호, CVC, 카드 비밀번호, 주민등록번호, 카드 유효기간 원문은 저장하지 않는다.

#### 반영 문서
- docs/policies/01_privacy-policy-draft.md
  - 2.5 결제 관련 정보

#### 개발 필요
- 카드 정보 입력은 PG 결제창 또는 PG 토큰화 방식 사용
- 서버/DB/log에 카드 원문이 남지 않도록 검토
- 결제 실패 로그에도 민감 카드 정보가 남지 않도록 처리

---

### 1.6 해지 후 업무데이터 다운로드는 제공하지 않는 방향으로 검토

#### 결정
해지 후 고객사에게 별도 데이터 다운로드 기간을 보장하지 않는 방향을 우선 검토한다.

#### 반영 문서
- docs/policies/01_privacy-policy-draft.md
  - 3.2 작업지시서 및 업무 데이터
- 향후 작성 예정:
  - docs/policies/02_terms-of-service-draft.md
  - docs/policies/04_storage-data-retention-policy-draft.md

#### 개발 필요
- 해지 전 데이터 백업/다운로드를 고객사가 직접 수행하도록 안내하는 기능 검토
- 해지 신청 전 경고 문구
- 해지 신청 전 “필요한 자료를 미리 저장해야 함” 안내
- 해지 후 접근 제한 정책
- 해지 취소 가능 기간 여부 결정

#### 주의
해지 후 다운로드를 제공하지 않는 것 자체가 항상 불가능한 것은 아니지만, 고객 분쟁 가능성을 줄이려면 해지 전 충분히 고지해야 한다. 특히 유료 SaaS에서 고객이 입력한 업무자료와 첨부파일이 있는 경우, 해지 전 안내와 약관 문구가 명확해야 한다.

---

### 1.7 첨부파일 휴지통 보관기간은 원칙적으로 30일

#### 결정
사용자가 삭제한 첨부파일 또는 삭제된 작업지시서 관련 파일은 원칙적으로 30일간 휴지통에 보관하고, 이후 영구 삭제될 수 있다고 표현한다.

#### 반영 문서
- docs/policies/01_privacy-policy-draft.md
  - 3.3 첨부파일 및 저장소 데이터
- 향후 작성 예정:
  - docs/policies/04_storage-data-retention-policy-draft.md

#### 개발 필요
- 삭제일 저장
- 휴지통 보관기간 30일 계산
- 영구 삭제 후보 표시
- 시스템관리자 purge 처리
- 영구 삭제 후 복구 불가 안내
- 고객사 관리자에게 남은 보관기간 표시 여부 검토

---

### 1.8 PG, 문자, 이메일 발송 서비스는 아직 미정

#### 결정
PG사, 문자 발송 서비스, 이메일 발송 서비스는 아직 확정하지 않는다. 이메일은 Google Workspace를 사용할 가능성이 있다.

#### 반영 문서
- docs/policies/01_privacy-policy-draft.md
  - 5.2 위탁사 확정 필요
  - 13. 내부 검토 필요 사항

#### 개발 필요

- 문자/이메일 발송 서비스를 사용할 경우 위탁사 반영
- Google Workspace를 단순 업무 이메일로 쓰는지, 서비스 자동 발송 인프라로 쓰는지 구분 필요
- 자동 이메일 발송이 필요하면 별도 발송 서비스 또는 Google API 사용 여부 검토

- 세금계산서 발행 상태까지 시스템에서 관리
- 외부 세금계산서 서비스 연동

#### 결정 필요 시점

## 4. P-0.91.4 추가 확정사항

### 4.1 국세청 API는 1차에 연동하지 않는다
- 고객사는 사업자명, 사업자등록번호, 대표자명, 사업자등록증 파일을 필수 입력한다.
- 시스템관리자는 승인 전 사업자등록증 파일을 직접 확인한다.
- 시스템관리자 화면에는 “사업자등록증 확인” 버튼을 둔다.
- 1차의 “사업자등록증 확인” 버튼은 API 호출이 아니라 수동 확인 상태를 기록하는 버튼이다.
- 개업일자 입력 UI는 1차에 넣지 않는다.
- API 상태조회 또는 진위확인은 추후 검토한다.

개발 필요:
- 사업자등록증 파일 필수 업로드
- 시스템관리자 승인 상세 화면에서 사업자등록증 파일 열람
- “사업자등록증 확인” 버튼
- 확인 완료 상태값, 확인 일시, 확인한 시스템관리자, 확인 메모

### 4.2 해지 취소는 다음 결제일 전날 23:59까지 가능
- 해지는 즉시 사용 종료가 아니라 다음 결제일부터 정기결제를 중단한다는 의미다.
- 해지 신청 후 다음 결제일 전까지는 정상 사용 가능하다.
- 이 기간에는 읽기 전용이 아니라 기존 기능을 정상 사용할 수 있다.
- 다음 결제일이 도래하면 해지 효력이 발생하고 서비스 접근이 제한된다.
- 해지 후 별도 읽기 전용 기간은 제공하지 않는다.
- 해지 후 별도 다운로드 기간은 제공하지 않는다.
- 해지 취소 시 구독 상태는 즉시 active 상태로 복귀한다.
- 해지 취소 버튼을 누르는 순간 추가 결제가 발생하는 것은 아니다.
- 다음 예정 결제일에 정기결제가 정상 재개된다.

개발 필요:
- 해지 신청 상태
- 다음 결제일
- 이용 종료 예정일
- 해지 취소 가능 기한
- 해지 신청/해지 취소
- 해지 안내 모달
- 로그인 시 남은 이용기간 팝업
- 이용 종료 후 접근 차단

### 4.3 데이터 내보내기 기능을 제공한다
- 버튼명은 “데이터 내보내기”로 한다.
- 고객사 관리자가 버튼을 누르면 안내 팝업을 표시한다.
- 팝업에는 백업 파일 생성 후 고객사 관리자 이메일로 다운로드 링크가 발송된다는 내용을 명시한다.
- 다운로드 링크는 생성 완료 후 7일 동안 유효하다는 내용을 명시한다.
- export zip은 서버에서 생성한다.
- 생성된 export zip은 임시 저장소에 저장한다.
- 이메일에는 zip 파일을 직접 첨부하지 않고 다운로드 링크만 발송한다.
- 7일 경과 후 export zip은 삭제될 수 있다.

개발 필요:
- 데이터 내보내기 버튼
- 안내 팝업
- export job 생성
- export zip 생성
- 임시 저장소 저장
- 다운로드 링크 생성
- 다운로드 가능 기한 7일
- 고객사 관리자 이메일 발송
- export zip 만료/삭제 처리

### 5.2 이메일 발송 방식은 Google Workspace 기반으로 표현한다
- 정책 문서에는 Gmail API 또는 SMTP 같은 구현 방식까지 명시하지 않는다.
- 초기 이메일 발송은 Google Workspace 기반으로 한다.
- 구체적인 발송 방식은 구현 시점에 기술 검토 후 정한다.
- 앱 구조는 추후 전문 발송 서비스로 교체 가능하게 설계한다.

개발 필요:
- sendEmail 또는 sendTransactionalEmail 추상화 모듈
- Google Workspace/Gmail API 방식 검토
- SMTP 방식 검토
- 발송 성공/실패 로그
- 발송 실패 시 관리자 확인 또는 재시도 기준

### 5.3 데이터 내보내기는 요청 기반 export job 방식으로 처리한다
1. 고객사 관리자가 “데이터 내보내기” 버튼을 클릭한다.
2. 안내 팝업을 확인한다.
3. export job이 생성된다.
4. 서버에서 export zip을 생성한다.
5. 생성 완료 후 고객사 관리자 이메일로 다운로드 링크를 발송한다.
6. 고객사 관리자는 로그인 후 다운로드한다.
7. 다운로드 가능 기간 7일이 지나면 export zip은 삭제될 수 있다.

개발 필요:
- export_jobs 테이블
- export 상태값: requested, processing, completed, failed, expired
- export zip 생성 작업
- R2 임시 저장
- 다운로드 링크 생성
- 완료 이메일 발송
- 만료 처리

### 5.4 다운로드 링크는 로그인 필요 방식으로 한다
- 이메일 링크 자체만으로는 파일을 다운로드할 수 없다.
- 다운로드 링크는 고객사 관리자 로그인 화면 또는 데이터 내보내기 다운로드 화면으로 이동시키는 용도로 사용한다.
- 로그인 후 고객사 관리자 권한을 확인한 뒤 다운로드를 허용한다.
- 다운로드 가능 기간은 생성 완료 후 7일이다.

개발 필요:
- 로그인 필요 다운로드 route
- 고객사 관리자 권한 확인
- export job 소유 회사 확인
- export job 요청자 또는 고객사 관리자 권한 확인
- 만료일 검사
- 다운로드 이력 기록

### 5.5 해지 신청 상태 안내 팝업은 로그인 세션 최초 1회 표시한다
- 해지 신청 상태인 고객사 관리자가 로그인하면 해당 로그인 세션에서 최초 1회 안내 팝업을 표시한다.
- 이용 종료일이 가까워진 경우 상단 배너 등 추가 고지를 검토한다.

안내 내용:
- 해지 신청 상태
- 이용 종료 예정일
- 다음 결제일부터 접근이 제한될 수 있음
- 필요한 자료는 이용 종료 전 데이터 내보내기를 통해 저장해야 함
- 데이터 내보내기 다운로드 링크는 7일간 유효함

- 고객사에는 안내 이메일/공지 중심으로 고지한다.
- 삭제 로그에는 고객사, 삭제 사유, 삭제 기준일, 삭제 처리일, 처리자, 삭제 대상 요약, 삭제 상태를 기록한다.

### 10.5 약관 변경 및 재동의 기준

- 요금

- 데이터 내보내기 제한
- 결제 실패/미납에 따른 서비스 제한

재동의가 필요 없는 변경:
- 오탈자 수정

## 12. P-0.91.12 추가 확정사항

### 12.1 고객사 휴지통 비우기

- 고객사 휴지통 비우기 또는 영구 삭제 요청은 삭제 로그에 기록한다.

### 12.2 고객사 휴지통과 시스템관리자 삭제관리 구분
- 고객사 휴지통은 복원 가능한 30일 보관 공간이다.
- 시스템관리자 삭제관리는 30일 경과 항목, 장기 미납 90일 항목, 해지/삭제 대상 데이터의 최종 purge 관리 공간이다.

### 12.3 휴지통 30일 경과 후 영구 삭제
- 고객사 휴지통에서 30일 지난 항목은 시스템관리자 화면의 “영구 삭제 가능 대상”으로 표시한다.
- 초기에는 자동 영구 삭제하지 않는다.
- 시스템관리자가 최종 확인 후 영구 삭제한다.
- 추후 안정화 후 자동 삭제를 검토할 수 있다.

### 12.4 시스템관리자 파일 접근
- 시스템관리자가 고객사 파일을 열람 또는 다운로드할 때 접근 사유 입력을 필수로 한다.
- 접근자, 접근일시, 고객사, 파일, 사유를 로그로 기록한다.
- 접근 사유는 다음 선택값을 둔다.
  - 고객지원 요청
  - 장애 대응
  - 보안 확인
  - 불법/부적절 파일 신고 대응
  - 장기 미납/해지/삭제 처리
  - 법령/기관 요청
  - 기타
- 기타 선택 시 직접 입력을 필수로 한다.

### 12.5 백업 복구 문구
- 백업은 서비스 장애 대응 및 운영 안정성을 위한 것이다.
- 회사는 고객사의 개별 삭제, 수정, 업로드 오류에 대한 복구를 항상 보장하지 않는다.
- 영구 삭제된 데이터는 복구가 제한되거나 불가능할 수 있다.
- 고객사는 중요한 자료를 서비스 이용기간 중 필요한 방식으로 별도 보관할 책임이 있다.

## 14. P-0.91.14 추가 확정사항

### 14.1 정기점검
- 정기점검은 특정 요일이나 시간으로 고정하지 않는다.
- 서비스 안정성, 보안, 기능 개선, 인프라 점검 등을 위해 필요 시 진행한다.

- 고객지원은 영업일 기준으로 순차 응대한다.
- 구체적인 응답 시간을 정책에 명시하지 않는다.
- 장애 또는 결제 관련 문의는 우선 확인할 수 있다.

### 14.3 부정 이용 대응
- 부정 이용은 원칙적으로 사전 안내 또는 시정 요청 후 제한한다.
- 다만 심각한 경우 사전 안내 없이 즉시 제한할 수 있다.

### 14.4 서비스 전체 종료
- 서비스 전체 종료는 WAFL 서비스를 더 이상 고객사에게 제공하지 않는 경우를 의미한다.
- 서비스 전체 종료 시 원칙적으로 최소 30일 전 사전 고지한다.
- 고객사는 고지 기간 중 데이터 내보내기를 이용할 수 있다.
- 천재지변, 법령상 요청, 보안상 긴급 사유, 외부 인프라 중단 등 긴급하거나 불가피한 경우 고지 기간이 단축될 수 있다.

### 14.5 장애 보상
- 서비스 장애에 대한 별도 금전 보상 또는 이용료 환불은 원칙적으로 제공하지 않는다.
- 다만 회사의 고의 또는 중대한 과실이 있거나 법령상 필요한 경우 회사가 정한 기준에 따라 예외적으로 조치할 수 있다.

- 사업자 정보, 시행일, PG사, 개인정보 보호책임자, 처리위탁사, 이메일 계정 실제 설정, 법무/세무 검토가 완료되기 전까지 최종 공개본으로 전환하지 않는다.

## 34. P-0.91.34 추가 확정사항

### 34.1 정책 기반 개발 우선순위
정책 기반 개발 우선순위를 다음 순서로 정리한다.

1. 정책 문서 공개 화면과 동의 이력
2. 고객사 승인 요청과 필수 정책 동의
3. 요금제, 무료 체험, 결제 실패 상태
4. 데이터 내보내기
5. 저장소, 휴지통, 영구 삭제 관리
6. 저장소 사용량과 제한
7. 장기 미납 관리
8. 정책 변경 재동의
9. 시스템관리자 파일 접근 로그
10. 결제/PG/영수증

### 34.2 즉시 개발 추천
정책 문서 작업 이후 코드 작업으로 돌아가면 고객사 관리자 “정책 및 약관” 화면과 동의 이력 저장부터 구현한다.
