# 정책 기반 개발 우선순위 정리

## 문서 상태
- Policy Version: P-0.91.34
- Based App Version: 0.15.73
- 문서 성격: 정책 문서와 실제 개발 작업 연결용
- 적용 대상: WAFL 정책 문서 기반 기능 개발 전체

## 1. 목적
이 문서는 WAFL 정책 문서에서 도출된 기능을 실제 개발 작업으로 연결하기 위한 우선순위를 정리한다.

## 2. 우선순위 판단 기준
1. 서비스 이용 시작에 필수인지
2. 고객 공개 문서와 실제 화면 동작 불일치 시 리스크가 큰지
3. 결제/해지/미납/데이터 삭제처럼 고객 피해 가능성이 큰지
4. 데이터 내보내기/휴지통/삭제처럼 고객 데이터 권리와 직접 연결되는지
5. 시스템관리자 감사/접근 로그처럼 분쟁 대응에 필요한지
6. 실제 PG/도메인/사업자등록 전에도 개발 가능한지

## 3. 개발 우선순위 요약

| 순위 | 개발 묶음 | 핵심 내용 | 선행 조건 |
| ---: | --- | --- | --- |
| 1 | 정책 문서 공개 화면과 동의 이력 | 정책 및 약관 메뉴, 문서 열람, 동의 이력 | 없음 |
| 2 | 고객사 승인 요청과 필수 정책 동의 | 승인 요청, 사업자등록증 필수, 약관 동의 | 없음 |
| 3 | 요금제/무료체험/결제 실패 상태 | Trial/Lite/Flow/Studio, 결제 실패 guard | PG 전 상태값 가능 |
| 4 | 데이터 내보내기 | export zip, 이메일 링크, 7일 만료 | 이메일 발송 필요 |
| 5 | 저장소/휴지통/영구 삭제 | 휴지통 30일, 복원, 비우기, purge | 저장소 구조 필요 |
| 6 | 저장소 사용량과 제한 | 80% 안내, 100% 업로드 제한 | 파일 업로드 흐름 |
| 7 | 장기 미납 관리 | 30/60/90일 안내와 삭제 가능 대상 | 결제 상태값 |
| 8 | 정책 변경 재동의 | 재동의 화면, 미동의 제한 | 동의 이력 |
| 9 | 시스템관리자 파일 접근 로그 | 접근 사유 필수, 로그 기록 | 파일 열람 route |
| 10 | 결제/PG/영수증 | PG 연동, billingKey, 영수증 URL | PG사 확정 |

## 4. 1순위 — 정책 문서 공개 화면과 동의 이력
정책 결정:
- 고객사 관리자 환경설정에 “정책 및 약관” 메뉴를 둔다.
- 일반 멤버도 이용약관과 개인정보처리방침은 열람할 수 있다.
- 정책 버전, 동의 일시, 사용자, 고객사, 재동의 여부를 기록한다.

필요 개발:
- 고객사 관리자 정책 및 약관 메뉴
- 정책 문서 목록/전문 보기
- 현재 적용 버전/시행일 표시
- 일반 멤버 열람 화면
- policy_versions
- policy_agreements

## 5. 2순위 — 고객사 승인 요청과 필수 정책 동의
정책 결정:
- 사업자등록증 파일은 승인 요청 시 필수다.
- 회사 로고/이미지는 선택이다.
- 승인 전 고객사는 업무 화면을 볼 수 없다.
- 사업자등록번호 API는 아직 붙이지 않고 “사업자등록증 확인” 버튼 UI만 준비한다.

필요 개발:
- 승인 요청 입력 화면
- 사업자등록증 파일 필수 업로드
- 회사 로고/이미지 선택 업로드
- 승인 요청 전 필수 약관 동의
- 승인 대기/거절/승인 상태 화면
- 시스템관리자 승인 검토 화면
- 사업자등록증 확인 버튼 UI

## 6. 3순위 — 요금제, 무료 체험, 결제 실패 상태
정책 결정:
- Trial 7일, 100MB, 멤버 3명
- Lite/Flow/Studio 유료 요금제
- 결제 실패 즉시 유료 업무 기능 제한
- 로그인, 결제수단 변경, 결제 재시도, 고객지원 문의, 정책 확인은 허용

필요 개발:
- company_subscription 상태
- plan_type
- trial_started_at / trial_ends_at
- billing_status
- payment_failed_at
- subscription_cancel_at
- current_period_ends_at
- 결제 실패 배너/팝업
- 유료 업무 기능 guard
- 요금제 선택 화면
- 저장소/멤버/export 제한 연결

## 7. 4순위 — 데이터 내보내기
정책 결정:
- 버튼명은 “데이터 내보내기”
- 서버에서 export zip 생성
- 생성 완료 후 고객사 관리자 이메일로 다운로드 링크 발송
- 다운로드 링크는 로그인 및 권한 확인 필요
- 다운로드 가능 기간은 7일
- 멤버정보는 이름, 전화번호, 역할만 포함

필요 개발:
- export_jobs
- export_status
- requested_by / company_id
- expires_at
- download_count
- R2 임시 저장 위치
- 로그인 필요 다운로드 route
- export 완료 이메일 템플릿
- export zip 만료 삭제
- 요금제별 export 횟수 제한
- 포함/제외 데이터 생성 로직

## 8. 5순위 — 저장소, 휴지통, 영구 삭제 관리
정책 결정:
- 고객사 휴지통 30일
- 고객사 휴지통 비우기 허용
- 30일 경과 데이터는 시스템관리자 영구 삭제 가능 대상으로 표시
- 초기 자동 영구 삭제 없음
- 시스템관리자가 최종 확인 후 삭제

필요 개발:
- deleted_at / deleted_by
- delete_reason_code
- purge_eligible_at
- purge_status
- 고객사 휴지통 화면
- 복원/선택 삭제/휴지통 비우기
- 강한 확인 모달
- 시스템관리자 영구 삭제 가능 대상 화면
- 삭제/복원/영구삭제 로그

## 9. 6순위 — 저장소 사용량과 제한
정책 결정:
- 저장소 80% 도달 시 안내
- 저장소 100% 초과 시 새 파일 업로드 제한
- 텍스트 작업지시서 생성, 메모 작성, 기존 데이터 조회, 파일 삭제, 휴지통 비우기, 요금제 업그레이드는 허용

필요 개발:
- company_storage_usage
- storage_used_bytes / storage_limit_bytes
- storage warning 계산
- 업로드 guard
- 80% 안내 배너
- 100% 업로드 제한 안내
- 파일 삭제 후 사용량 재계산

## 10. 7순위 — 장기 미납 관리
정책 결정:
- 미납 30일: 장기 미납 안내
- 미납 60일: 데이터 삭제 예정 안내
- 미납 90일: 시스템관리자 삭제 가능 대상으로 표시
- 자동 삭제하지 않음

필요 개발:
- unpaid_since
- unpaid_30d_notice_sent_at
- unpaid_60d_notice_sent_at
- unpaid_90d_eligible_at
- 장기 미납 대상 목록
- 30일/60일 이메일 및 팝업 템플릿
- 90일 삭제 가능 대상 표시
- 시스템관리자 최종 삭제 처리

## 11. 8순위 — 정책 변경 재동의
정책 결정:
- 요금/환불/해지/데이터삭제/개인정보/데이터 내보내기 제한/결제 실패·미납 제한 변경은 재동의 대상이다.
- 고객지원 문의와 해지 신청은 재동의 전에도 허용한다.

필요 개발:
- policy_versions.requires_reagreement
- re_agreement_required 상태 계산
- 로그인 후 재동의 화면 선표시
- 재동의 미완료 시 업무 화면 접근 제한
- 고객지원/해지 예외 허용
- 정책 변경 이력 요약 표시

## 12. 9순위 — 시스템관리자 파일 접근 로그
정책 결정:
- 파일 열람/다운로드 시 접근 사유 입력 필수
- 접근자, 접근일시, 고객사, 파일, 사유 로그 기록

필요 개발:
- admin_file_access_logs
- 접근 사유 모달
- 기타 직접입력 필수
- 시스템관리자 파일 미리보기/다운로드 guard
- 감사로그 연동

## 13. 10순위 — 결제/PG/영수증
정책 결정:
- 카드결제 중심
- PG사 미정
- 세금계산서는 원칙적으로 별도 발행하지 않음
- 영수증/카드매출전표는 PG사 또는 카드사를 통해 제공

필요 개발:
- PG사 확정 후 결제 연동
- billing_key/customer_key 저장
- 결제 이력
- 영수증 URL
- 결제 실패 사유
- 환불 수동 처리 기록
- 결제수단 변경
- 정기결제 재시도
- 카드 마지막 4자리/카드사명 표시

## 14. 즉시 다음 코드 작업 추천
정책 문서 작업 이후 실제 코드 작업으로 돌아간다면 다음 순서를 추천한다.

1. 고객사 관리자 환경설정에 “정책 및 약관” 메뉴 추가
2. docs/policies/public 문서를 앱 화면에서 열람 가능하게 route 구성
3. 일반 멤버도 이용약관/개인정보처리방침 열람 가능하게 처리
4. policy_versions/policy_agreements 설계
5. 고객사 승인 요청 시 필수 약관 동의 연결
6. 사업자등록증 필수 업로드 UI/검증 연결

## 15. 판단
정책 문서 작업은 현재 단계에서 일단 종료 가능하다.

아직 고객 공개 v1.0 최종본으로 전환할 수는 없지만, 실제 개발로 연결할 기준은 충분히 정리되었다.
