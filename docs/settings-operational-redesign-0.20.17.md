# 환경설정 운영형 UI 2차 재설계 - 0.20.17

## 목적

0.20.11~0.20.16에서 환경설정 화면의 밀도와 카드형 설명을 1차로 줄였지만, 실제 운영 화면 관점에서는 아직 몇 가지 흐름이 임시적이다. 이 문서는 다음 구현 버전에서 환경설정을 “설명 카드 모음”이 아니라 “실제 운영 액션 중심 화면”으로 정리하기 위한 기준을 확정한다.

## 공통 원칙

- 설명 카드보다 실제 데이터, 목록, 문서, 입력 액션을 먼저 보여준다.
- 상시 노출 뱃지는 상태 판단에 필요한 것만 사용한다.
- 운영 기준, 요청 흐름, 후속 연결 같은 내부 설명은 고객사 관리자 화면에서 기본 노출하지 않는다.
- 요청은 “해당 필드/항목에서 바로 요청”하는 구조를 우선한다.
- 이력은 하단 목록으로 유지하되, 이력 자체가 주 화면보다 커지지 않게 한다.
- 임시 이메일 작성 흐름은 고객 화면에 노출하지 않고, DB 저장형 문의 접수 흐름으로 설계한다.
- 모달은 실제 문서 열람, 필드별 변경 요청, 유형 추가 요청처럼 사용자가 현재 작업을 완료하기 위해 필요한 경우에만 사용한다.

## 1. 약관·정책 탭

### 현재 문제

- 화면에는 5개 문서만 보이지만 `docs/정책문서/고객공개/`에는 더 많은 Markdown 문서가 존재한다.
- 현재 모달은 문서 내용을 실제 파일에서 읽는 느낌이 약하고, 문서 열람 화면으로 보기에는 완성도가 낮다.
- 고객은 문서 요약 카드가 아니라 실제 약관/정책 원문을 읽어야 한다.

### 방향

- `docs/정책문서/고객공개/`의 Markdown 문서를 실제 문서 원본으로 사용한다.
- 보기 버튼을 누르면 Markdown 내용을 읽어 문서형 모달로 렌더링한다.
- 모달은 제목, 버전, 시행일, 필수 여부, 본문을 분리해서 보여준다.
- 모달 본문은 카드 조각이 아니라 문서형 흐름으로 스크롤된다.
- 문서 열람 모달에는 닫기 버튼을 상단/하단에 모두 제공한다.

### 5개 노출 vs 8개 노출 기준

고객 공개 폴더의 파일 수와 화면 노출 문서 수를 반드시 1:1로 맞추지는 않는다. 고객이 직접 선택해야 하는 대표 문서는 5개를 유지하고, 나머지는 대표 문서의 내부 섹션 또는 운영 보조 문서로 분류한다.

대표 노출 후보:

1. 이용약관
2. 개인정보처리방침
3. 요금·환불정책
4. 데이터 보관·삭제정책
5. 서비스 운영정책

보조 문서 후보:

- 데이터 내보내기 정책
- 저장소 보관 정책
- 정책 고지·동의 정책

보조 문서는 필요 시 대표 문서 안의 관련 섹션으로 연결하거나, “세부 정책 보기” 형태로 접는다. 고객 화면에서 8개를 모두 병렬 노출하면 문서 체계가 과해 보일 수 있으므로 1차 구현은 5개 대표 문서 기준으로 한다.

## 2. 서비스 건의 탭

### 현재 문제

- 이메일 작성 방식은 임시 연결처럼 보인다.
- `peacebypiece.feedback@example.com` 같은 내부 이메일 주소가 화면에 노출되어 제품 완성도가 낮아 보인다.
- 실제 운영에서는 DB에 문의를 저장하고 시스템관리자가 처리해야 한다.

### 방향

- “이메일 작성하기”를 “문의하기” 또는 “접수하기”로 바꾼다.
- 이메일 주소는 고객 화면에서 제거한다.
- 문의 유형, 제목, 내용, 첨부 영역을 폼으로 제공한다.
- 제출하면 DB에 `service_feedback_requests` 같은 구조로 저장하는 방향을 전제로 한다.
- API가 아직 없을 경우에도 UI 문구는 DB 접수형으로 유지하고, 실제 저장 구현은 후속 버전으로 분리한다.
- 우측 영역은 이메일 안내가 아니라 최근 접수 이력과 처리 상태를 표시한다.

권장 상태값:

- 접수됨
- 검토중
- 답변완료
- 반려 또는 보류

## 3. 회사 정보 탭

### 현재 문제

- “회사 정보 변경 요청” 버튼을 누른 뒤 하단 작성 영역에 입력하는 흐름은 사용자가 어떤 필드를 바꾸는지 직관적이지 않다.
- 회사명, 사업자등록번호, 주소 같은 필드별 변경 요청이 더 자연스럽다.

### 방향

- 회사명, 사업자등록번호, 회사 주소, 대표 연락처 같은 필드 옆에 작은 수정/요청 아이콘을 둔다.
- 필드 아이콘을 누르면 해당 필드 전용 변경 요청 모달을 연다.
- 모달에는 현재 값, 변경 요청 값, 요청 사유를 표시한다.
- 요청 제출 후 하단 요청 이력에 누적한다.
- 전체 하단 작성 폼은 제거하거나, “기타 회사 정보 변경 요청”으로 축소한다.
- 탈퇴 요청은 별도 주요 액션으로 유지한다.

### 탈퇴 요청

- “계정 비활성화 요청”보다 고객 표현은 “탈퇴 요청”이 적합하다.
- 탈퇴 요청은 별도 모달로 확인한다.
- 시스템관리자 승인 후 사용자가 로그인하면 안내 모달 또는 차단 화면으로 남은 접근 가능 기간과 파일 다운로드 필요성을 안내한다.

## 4. 기준정보 설정 탭

### 현재 문제

- 내부 탭 구조는 적절하지만 “선택 항목 관리” 모달로 다시 들어가는 구조는 불필요하다.
- 생산품 유형, 단위 표준, 외주공정 유형은 같은 기준정보처럼 보이지만 운영 권한이 다르다.
- 코드명, 순서, 반복 뱃지는 고객사 관리자에게 필요성이 낮다.

### 방향

- 기준정보 탭 안에서 바로 목록을 관리한다. 별도 상위 관리 모달은 제거한다.
- 생산품 유형은 고객사가 직접 추가, 수정, 사용/미사용 전환을 할 수 있다.
- 단위 표준과 외주공정 유형은 시스템 공통 기준으로 보고 고객사는 추가 요청만 할 수 있다.
- 단위 표준/외주공정 유형 탭에는 “유형 추가 요청” 버튼을 제공한다.
- 요청 버튼을 누르면 요청 사유와 제안 이름을 입력하는 모달을 연다.
- 시스템관리자가 승인하면 공통 기준에 반영되는 흐름으로 본다.

### 표시 기준

제거 또는 숨김 대상:

- 코드명
- 순서
- 반복되는 녹색 상태 뱃지
- 사용 3 / 전체 3 같은 과한 통계

유지 대상:

- 이름
- 사용 여부
- 수정/요청 액션

사용 여부는 WAFL 스타일의 부드러운 토글 또는 pill control로 표시한다. 단순한 “사용” 녹색 뱃지 반복보다 토글형 컨트롤이 설정 화면에 더 적합하다.

## 자동테스트 기준

- 약관·정책: 보기 버튼 클릭 시 문서 모달이 열리고 실제 문서명이 표시되어야 한다.
- 서비스 건의: 제목/내용 입력 후 문의하기 버튼이 활성화되어야 한다.
- 회사 정보: 특정 필드의 수정 아이콘을 누르면 해당 필드 변경 요청 모달이 열려야 한다.
- 기준정보: 생산품 유형은 직접 관리 액션이 보이고, 단위 표준/외주공정 유형은 요청 액션이 보여야 한다.

## 이번 버전 범위

0.20.17은 설계 기준 문서화와 버전 업데이트만 포함한다. 기능 코드, DB schema, API, R2, 첨부/메모/휴지통/purge 흐름은 변경하지 않는다.
