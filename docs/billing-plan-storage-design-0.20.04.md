# 요금제/무료체험/저장공간 운영 데이터 설계 1차 (0.20.04)

## 1. 목적

`0.20.04`는 실제 결제 연동 전에 WAFL의 요금제, 무료체험, 저장공간, 멤버 제한, 결제 실패/해지/미납 상태를 일관된 운영 데이터 구조로 정리하는 설계 버전이다.

이번 버전은 문서 설계만 포함한다. DB schema, API route, PG 연동, R2 업로드 제한, UI 변경은 후속 버전에서 진행한다.

## 2. 범위

포함한다.

- 무료체험 7일 기준
- 무료체험 저장공간 100MB 기준
- 무료체험 멤버 3명 제한 기준
- Lite / Flow / Studio 요금제 후보
- 결제 실패, 해지 예정, 해지 완료, 미납, 일시정지 상태
- `company_subscriptions` 1차 데이터 구조 후보
- 저장공간 사용량 집계 기준
- 멤버 제한 적용 기준
- 후속 자동테스트 기준

포함하지 않는다.

- 실제 PG 연동
- 결제수단 저장
- 카드 승인/취소/환불 처리
- 세금계산서/현금영수증 처리
- R2 업로드 차단 구현
- 요금제 변경 UI 구현

## 3. 운영 원칙

요금제 상태는 고객사 단위로 관리한다. 사용자 개인이 아니라 `company_id` 기준으로 현재 요금제, 무료체험 종료일, 저장공간 한도, 멤버 한도를 판단한다.

초기에는 PG와 결제수단을 저장하지 않는다. 운영자가 수동으로 요금제 상태를 조정할 수 있는 구조를 먼저 확보하고, 실제 PG 연동은 별도 버전에서 붙인다.

업무 데이터 보존과 서비스 접근 제한은 분리한다. 결제 실패나 미납 상태가 되더라도 기존 작업지시서, 첨부, 메모, 회사 정보는 삭제하지 않는다. 제한은 신규 업로드, 신규 멤버 초대, 일부 업무 생성 기능부터 단계적으로 적용한다.

## 4. 요금제 후보

### 4.1 Trial

무료체험 상태다. 고객사 승인 후 첫 사용 시점 또는 승인 완료 시점을 기준으로 7일을 부여한다.

기본 기준은 다음과 같다.

- 기간: 7일
- 저장공간: 100MB
- 멤버 수: 고객사 관리자 포함 3명
- 기능: 핵심 업무 흐름 확인 가능
- 결제수단: 필수 아님

### 4.2 Lite

초소형 팀 또는 초기 샘플실용 기본 유료 요금제다.

후보 기준은 다음과 같다.

- 저장공간: 1GB
- 멤버 수: 5명
- 작업지시서/협력업체/기준정보 기본 기능 포함
- 고급 통계, 대량 파일, 자동화 기능은 제한 가능

### 4.3 Flow

일반 운영 팀 기준의 표준 요금제다.

후보 기준은 다음과 같다.

- 저장공간: 10GB
- 멤버 수: 15명
- 작업지시서, 협력업체, 원단·부자재 발주, 저장소, 통계 기본 포함
- 운영 리포트와 권한 관리 확장 가능

### 4.4 Studio

파일 사용량이 많고 여러 담당자가 협업하는 고객사용 상위 요금제다.

후보 기준은 다음과 같다.

- 저장공간: 50GB 이상
- 멤버 수: 50명 이상
- 고급 통계, 감사 로그, 대량 파일, 운영 지원 옵션 확장 가능
- 별도 계약 또는 수동 견적 적용 가능

## 5. 구독 상태 코드

1차 상태 코드는 다음을 기준으로 한다.

| 상태 코드 | 의미 | 업무 접근 기준 |
|---|---|---|
| `trialing` | 무료체험 중 | 정상 사용 |
| `active` | 유료 사용 중 | 정상 사용 |
| `past_due` | 결제 예정일 경과/확인 필요 | 경고 표시, 일정 기간 정상 사용 |
| `payment_failed` | 결제 실패 | 신규 업로드/초대 제한 후보 |
| `cancel_scheduled` | 해지 예정 | 다음 결제일까지 정상 사용 |
| `canceled` | 해지 완료 | 조회/내보내기 중심 제한 후보 |
| `suspended` | 운영자 정지 | 업무 접근 제한 |

`past_due`와 `payment_failed`는 초기에 운영자 수동 조정으로 처리한다. 자동 PG webhook 반영은 후속 결제 연동 단계에서 붙인다.

## 6. company_subscriptions 후보 구조

`0.20.05`에서 추가할 수 있는 1차 테이블 후보는 다음과 같다.

```sql
create table company_subscriptions (
  id uuid primary key default gen_random_uuid(),
  company_id uuid not null references companies(id) on delete cascade,
  plan_code text not null,
  status text not null,
  trial_started_at timestamptz,
  trial_ends_at timestamptz,
  current_period_started_at timestamptz,
  current_period_ends_at timestamptz,
  cancel_scheduled_at timestamptz,
  canceled_at timestamptz,
  storage_limit_bytes bigint not null,
  member_limit integer not null,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  constraint company_subscriptions_plan_code_check
    check (plan_code in ('trial', 'lite', 'flow', 'studio', 'custom')),
  constraint company_subscriptions_status_check
    check (status in ('trialing', 'active', 'past_due', 'payment_failed', 'cancel_scheduled', 'canceled', 'suspended'))
);
```

초기에는 회사당 활성 구독 1개 원칙으로 운용한다. 과거 이력 테이블은 후속 단계에서 분리할 수 있다.

## 7. 저장공간 사용량 기준

저장공간 사용량은 고객사 단위로 계산한다.

포함 후보는 다음과 같다.

- 작업지시서 첨부 문서
- 디자인 파일
- 회사 대표 이미지
- 사업자등록증 파일
- 기타 R2에 저장되는 고객사 소유 파일

제외 후보는 다음과 같다.

- DB 텍스트 데이터
- 메모 텍스트
- 정책 문서
- 시스템 공용 리소스
- 삭제 완료된 파일

휴지통 보관 중인 파일은 실제 R2 저장공간을 점유하므로 포함하는 쪽이 더 현실적이다. 단, 고객 안내에서는 “휴지통 파일도 저장공간에 포함될 수 있음”을 명시해야 한다.

## 8. 멤버 제한 기준

멤버 수는 고객사 관리자와 활성 멤버를 포함한다.

포함한다.

- 고객사 관리자
- 승인 완료 일반 멤버
- 활성 상태 멤버

제외한다.

- 시스템관리자
- 승인 대기 초대
- 거절된 초대
- 탈퇴 완료 멤버
- 비활성/정지 멤버는 운영 정책에 따라 포함 여부를 후속 확정

초기 구현에서는 “활성 사용자” 기준으로 제한하는 것이 관리가 쉽다. 비활성 멤버를 한도에 포함할지는 요금 정책 문구 확정 후 결정한다.

## 9. 제한 UX 기준

저장공간 제한은 다음 단계로 설계한다.

- 80% 이상: 경고 표시
- 100% 이상: 신규 파일 업로드 제한
- 100% 이상에서도 허용: 조회, 삭제, 휴지통 비우기, 텍스트 메모, 작업지시서 조회
- 100% 이상에서 제한 후보: 첨부 업로드, 디자인 업로드, 회사 파일 업로드

멤버 제한은 다음 단계로 설계한다.

- 한도 도달 전: 멤버관리 화면에 현재 사용량 표시
- 한도 도달: 신규 초대/승인 제한
- 한도 초과 상태: 기존 멤버 강제 삭제 없이 추가만 제한

## 10. 고객사 화면 표시 기준

환경설정에는 다음 정보를 표시한다.

- 현재 요금제
- 상태
- 무료체험 종료일 또는 현재 결제 기간 종료일
- 저장공간 사용량/한도
- 멤버 수/한도
- 제한 또는 경고 상태
- 요금제 변경 요청 안내

초기에는 결제 화면이 아니라 “운영팀 문의/변경 요청” 형태로 둔다.

## 11. 시스템관리자 화면 표시 기준

시스템관리자 고객사 상세에는 다음 정보를 표시한다.

- 현재 요금제
- 구독 상태
- 무료체험 종료일
- 저장공간 한도
- 멤버 한도
- 상태 수동 변경 버튼 후보
- 운영 메모 후보

실제 결제 이력, PG 거래번호, 카드 정보는 이번 단계에서 제외한다.

`0.20.05` 이후에는 다음 테스트를 추가한다.

- `company_subscriptions` required table/columns smoke
- 기본 trial subscription fixture 생성/조회
- plan code/status check constraint 검증
- 저장공간 limit/member limit 조회 API 검증
- 환경설정 요금제 UI E2E mock 검증

## 14. 현재 버전 결론

`0.20.04`는 기능 구현 전 운영 데이터 설계 확정 단계다. DB schema, API, UI, R2 upload guard는 변경하지 않는다. 다음 버전 `0.20.05`에서 `company_subscriptions` 테이블과 현재 고객사 구독 상태 조회 API를 추가한다.
