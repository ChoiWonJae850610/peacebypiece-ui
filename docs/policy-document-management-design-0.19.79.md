# 정책/약관 고객 공개 문서 연결 설계 1차 (0.19.79)

## 목적

WAFL 고객 공개 문서와 내부 운영 문서를 분리하고, 이후 정책 버전/동의 이력 DB 구현으로 연결할 기준을 정리한다.

현재 단계는 설계 문서 추가와 앱 버전 갱신만 포함한다. 실제 DB table, API route, 공개 화면 구현은 포함하지 않는다.

## 공개 문서 분류

### 고객 공개 필수 문서

- 이용약관
- 개인정보처리방침
- 요금 및 환불 정책
- 요금제 및 저장소 정책
- 저장소 보관 및 삭제 정책
- 데이터 내보내기 정책
- 서비스 운영 정책
- 정책 변경 고지 및 재동의 정책

### 고객 공개 선택 문서

- 고객지원 안내
- 장애 및 점검 안내 기준
- 보안 안내
- 파일 보관/접근 안내

### 내부 운영 문서

- 시스템관리자 운영 매뉴얼
- 승인/반려 처리 기준
- 감사로그 검토 기준
- 결제 실패/미납 운영 기준
- 고객사 해지 및 데이터 삭제 처리 기준

내부 운영 문서는 고객 화면에 직접 노출하지 않는다. 고객 공개 문서로 전환할 문장은 별도 고객 공개 문체로 정리한다.

## 고객 공개 화면 구조

### 고객사 관리자

위치 후보:

- `/workspace/settings` > 정책 및 약관
- 고객사 관리자 환경설정 카드 내 문서 목록

표시 항목:

- 문서명
- 적용 버전
- 시행일
- 마지막 개정일
- 필수 동의 여부
- 전문 보기 버튼

### 일반 멤버

위치 후보:

- 개인 설정
- 앱 하단 정책 링크

표시 항목:

- 이용약관
- 개인정보처리방침
- 서비스 운영 정책
- 데이터 내보내기/보관 안내 중 멤버에게 필요한 범위

### 비로그인 또는 승인 전 사용자

위치 후보:

- 로그인/초대/승인 요청 화면 하단
- 승인 요청 전 동의 화면

표시 항목:

- 이용약관
- 개인정보처리방침
- 요금 및 환불 정책
- 서비스 운영 정책

## DB 설계 후보

### policy_documents

정책 문서의 논리 단위를 저장한다.

후보 컬럼:

- id
- code
- title
- audience
- category
- is_required
- is_public
- display_order
- created_at
- updated_at

### policy_versions

정책 문서의 버전별 본문과 시행일을 저장한다.

후보 컬럼:

- id
- policy_document_id
- version
- effective_date
- published_at
- content_markdown
- summary
- requires_reagreement
- created_by_system_user_id
- created_at
- updated_at

### policy_agreements

사용자 또는 고객사의 동의 이력을 저장한다.

후보 컬럼:

- id
- policy_version_id
- company_id
- user_id
- agreed_at
- agreement_context
- ip_address_hash
- user_agent_hash
- created_at

개인정보 최소화 원칙상 IP와 user agent는 원문 저장보다 hash 또는 별도 보관 정책을 검토한다.

## 동의 흐름

### 최초 고객사 승인 요청

1. 초대 링크 진입
2. 회사 정보 입력
3. 사업자등록증 업로드
4. 필수 정책 확인
5. 필수 동의 체크
6. 승인 요청 제출
7. policy_agreements 저장

### 정책 중요 변경

1. 시스템관리자가 새 정책 버전 게시
2. requires_reagreement = true인 문서가 있으면 재동의 대상 생성
3. 로그인 후 재동의 화면 우선 표시
4. 미동의 시 업무 화면 접근 제한
5. 고객지원/해지/데이터 내보내기 안내는 예외 접근 검토

## 화면 구현 우선순위

1. 고객사 관리자 정책 문서 목록 및 전문 보기
2. 일반 멤버 정책 열람
3. 정책 버전 DB/API
4. 승인 요청 필수 동의 연결
5. 정책 변경 재동의
6. 시스템관리자 정책 문서 관리 화면
